## 二、总体流程

```text
前端点击启动
-> api.js:startSimulation(payload) 发送 POST /control/start
-> main.py:control_start(...) 设置系统状态为 RUNNING
-> simulation.py:_loop() 开始循环调用 step()
-> simulation.py:_step() 依次执行 robot_phase、controller_phase、navigator_phase
-> Robot 扫描地图或移动小车
-> Controller 分配探索目标
-> Navigator 规划路径
-> Redis 黑板保存地图、任务、路径等状态
-> main.py:broadcast_loop() 生成 STATE_SNAPSHOT
-> ws.js:createSimulationSocket(...) 收到推送
-> VisualizerCanvas.vue 刷新 3D 画面
```

## 三、页面初始化：先拿状态，再连 WebSocket

前端页面主体在 `PathfindingVisualizer.vue`。组件引入 HTTP 方法和 WebSocket 方法的位置是 `PathfindingVisualizer.vue:254-269`，这里引入了 `fetchState()`、`startSimulation()`、`createSimulationSocket(...)` 等方法。

页面准备好 3D 地面后会执行 `PathfindingVisualizer.vue:470-486` 的 `initBackend()`。这里做两件事：

1. `PathfindingVisualizer.vue:479` 调用 `api.js:86-88` 的 `fetchState()`。
2. `PathfindingVisualizer.vue:481-486` 调用 `ws.js:4-53` 的 `createSimulationSocket(...)` 建立 WebSocket。

`fetchState()` 实际请求后端：

```text
GET /state
```

后端接收位置是 `main.py:331-336` 的 `get_state(...)`，它会调用 `main.py:266-272` 的 `snapshot_with_runtime(...)` 返回当前地图、小车、任务、路径等状态。

WebSocket 连接由 `ws.js:9-10` 的内部方法 `connect()` 发起，地址是 `/ws?token=...`。收到消息后，`ws.js:16-21` 的 `socket.onmessage` 判断消息类型是不是 `STATE_SNAPSHOT`，是的话就调用 `onSnapshot(message.payload)`。

## 四、用户点击启动探索

启动按钮绑定在 `PathfindingVisualizer.vue:42-50`，点击后调用 `PathfindingVisualizer.vue:568-578` 的 `handleBackendStart()`。这个方法会：

1. 必要时调用 `api.js:131-136` 的 `setVehicles(payload)` 更新小车数量。
2. 调用 `api.js:94-99` 的 `startSimulation(payload)`。

`startSimulation(payload)` 发送：

```text
POST /control/start
body: { policy, navigatorAlgorithm }
```

后端接口是 `main.py:601-615` 的 `control_start(...)`，核心动作是：

- `main.py:607-610`：调用 `configure_policy(...)` 和 `configure_navigator_algorithm(...)` 设置策略与路径算法。
- `main.py:611`：调用 `simulation.ensure_demo_vehicles()` 确保小车存在。
- `main.py:612`：调用 `blackboard.set_system_status("RUNNING")`。
- `main.py:613-614`：调用 `snapshot_with_runtime()` 生成快照，并调用 `replay_store.start(...)` 开始记录。

所以前端点击启动只是发命令，真正的小车探索从后端状态变成 `RUNNING` 后才开始。

## 五、后端仿真循环如何跑起来

后端启动时会执行 `main.py:290-297` 的 `on_startup()`：

- `main.py:294` 调用 `simulation.py:69-72` 的 `start_background()`，启动仿真循环。
- `main.py:297` 调用 `asyncio.create_task(broadcast_loop())`，启动 WebSocket 广播循环。

仿真循环本身在 `simulation.py:83-88` 的 `_loop()`：

```text
如果 blackboard.system_status == RUNNING
-> 调用 self.step()
-> 等待 0.45 秒
-> 下一轮继续
```

单轮调度由 `simulation.py:277-306` 完成：

1. `simulation.py:277-280` 的 `step()` 负责加锁并调用 `_step()`。
2. `simulation.py:281-291` 的 `_step()` 决定本轮执行顺序。
3. `simulation.py:293-295` 的 `robot_phase()` 调用 Robot。
4. `simulation.py:299-302` 的 `controller_phase()` 调用 Controller。
5. `simulation.py:304-306` 的 `navigator_phase()` 调用 Navigator。

## 六、Robot：扫描地图或移动小车

Robot 组件创建位置是 `simulation.py:34`，实现文件是 `component.py（robot）`。

Robot 每轮入口是 `component.py（robot）:42-47` 的 `run_once()`，它遍历所有小车，并逐个调用 `component.py（robot）:49-99` 的 `run_vehicle_once(...)`。

`run_vehicle_once(...)` 的核心分支：

- `component.py（robot）:56-61`：没有任务时，调用 `scan_and_upload(...)` 扫描地图。
- `component.py（robot）:63-65`：任务等待规划或规划失败时，继续扫描。
- `component.py（robot）:67-99`：任务已有 `pathQueue` 时，取下一步位置并更新小车状态。
- `component.py（robot）:81-84`：下一步被真实障碍挡住时，调用 `report_blocked(...)` 等待重新规划。

扫描入口是 `component.py（robot）:138-142` 的 `scan_and_upload(...)` 和 `detect_frontiers(...)`，最终进入 `frontier_scan.py`。

## 七、Frontier：扫描地图并发现候选目标

扫描地图的核心代码是 `frontier_scan.py:35-73` 的 `scan_and_upload(...)`：

- `frontier_scan.py:40-60`：以小车当前位置为中心，按扫描半径生成周围格子状态。
- `frontier_scan.py:61-69`：调用 `base.py:930-970` 的 `upload_map_patch(...)`，把地图 patch 合并进 Redis 黑板。
- `frontier_scan.py:70-73`：需要发现 frontier 时，调用 `detect_frontiers(...)`。

frontier 发现逻辑是 `frontier_scan.py:75-112` 的 `detect_frontiers(...)`：

- `frontier_scan.py:81-89`：找“已知可走且旁边还有未知区域”的点。
- `frontier_scan.py:91-99`：计算未知区域收益并排序。
- `frontier_scan.py:100-110`：调用 `base.py:972-997` 的 `save_frontier(...)` 保存 frontier。

## 八、Controller：给空闲小车分配 frontier

Controller 组件创建位置是 `simulation.py:35-39`，每轮调用位置是 `simulation.py:299-302` 的 `controller_phase()`。具体实现是 `component.py（controller）:24-66` 的 `run_once()`。

`run_once()` 的工作分三步：

1. `component.py（controller）:27-31`：调用 `base.py:1261-1266` 的 `pending_tasks_without_request()`，给等待重新规划的任务补导航请求。
2. `component.py（controller）:33-40`：读取快照、开放 frontier 和空闲小车，其中开放 frontier 来自 `base.py:1240-1251` 的 `open_frontiers()`，空闲小车来自 `base.py:1253-1259` 的 `idle_vehicles()`。
3. `component.py（controller）:43-64`：调用策略的 `select_assignments(...)`，再调用 `base.py:1040-1060` 的 `create_task_for_frontier(...)` 和 `base.py:1062-1083` 的 `create_navigation_request(...)`。

策略代码有两类：

- 最近可达策略：`nearest_reachable_frontier.py:20-84` 的 `select_assignments(...)` 生成分配结果，`nearest_reachable_frontier.py:86-135` 的 `nearest_reachable_frontier(...)` 选择最近且可达的 frontier。
- 贪心收益策略：`greedy_frontier.py:17-53` 的 `select_assignments(...)` 生成分配结果，`greedy_frontier.py:55-79` 的 `choose_frontier(...)` 按“未知区域收益 - 距离代价”选择 frontier。

## 九、Navigator：为任务规划路径

Navigator 组件创建位置是 `simulation.py:40`，每轮调用位置是 `simulation.py:304-306` 的 `navigator_phase()`。具体实现是 `component.py（navigator）`。

普通规划流程：

1. `component.py（navigator）:20-32` 的 `run_once()` 调用 `base.py:1107-1123` 的 `claim_navigation_request(...)` 领取一个导航请求。
2. `component.py（navigator）:55-66` 的 `plan_request(...)` 调用 `factory.py:9-12` 的 `create_planner(...)` 创建规划器，并执行 `planner.plan(...)`。
3. `component.py（navigator）:68-112` 的 `plan_from_result(...)` 把路径整理成 navigation plan。
4. `component.py（navigator）:31` 调用 `base.py:1146-1171` 的 `write_navigation_plan(...)` 写回 Redis 黑板。

CBS 批量规划流程在 `component.py（navigator）:33-54` 的 `run_cbs_once()`，它会调用 `base.py:1125-1144` 的 `claim_navigation_requests(...)` 一次领取多个请求，并调用 `CBSPlanner`。

主要算法位置如下：

- `astar_planner.py:10-68` 的 `astar(...)` 和 `astar_planner.py:71-78` 的 `AStarPlanner.plan(...)`：普通 A*。
- `st_astar_planner.py:10-91` 的 `st_astar(...)`：时空 A*。
- `cbs_planner.py:18-95` 的 `CBSPlanner.plan_batch(...)`：CBS 多车冲突协调。
- `cbs_planner.py:97-137` 的 `low_level_search(...)`：CBS 底层搜索，最终仍调用 `st_astar(...)`。

规划成功后，`base.py:1146-1171` 的 `write_navigation_plan(...)` 会把任务状态改为 `PLANNED`，并把路径写入任务的 `pathQueue`。Robot 下一轮就会通过 `base.py:1173-1178` 的 `get_vehicle_task(...)` 读取任务并沿路径移动。

## 十、后端生成并推送状态快照

后端状态快照由 `main.py:266-272` 的 `snapshot_with_runtime(...)` 生成，里面调用 `blackboard.snapshot_since(...)` 并追加 runtime 信息。

Redis 快照读取位置是 `blackboard.py:680-779` 的 `snapshot_since(...)`，包含：

```text
map / mapDelta
vehicles
frontiers
tasks
navigationRequests
navigationPlans
heartbeats
events
systemStatus
```

WebSocket 接口是 `main.py:647-657` 的 `websocket_endpoint(...)`。前端连上后，后端会先发一次：

```text
{ type: "STATE_SNAPSHOT", payload: initial_snapshot }
```

持续推送由 `main.py:695-718` 的 `broadcast_loop()` 完成：后端定期调用 `snapshot_with_runtime(...)` 生成新快照，并通过 `websocket.send_json(...)` 发给所有在线前端。

## 十一、前端收到快照后如何刷新 3D 画面

前端 WebSocket 收到 `STATE_SNAPSHOT` 的位置是 `ws.js:16-21` 的 `socket.onmessage`。它把 payload 回调给 `PathfindingVisualizer.vue:493-497` 的 `applyBackendSnapshot(snapshot)`。

前端状态保存位置是 `simulationStore.js:12-17` 的 `applySnapshot(snapshot, previousGridOverride)`：

- `simulationStore.rawSnapshot` 保存后端原始快照。
- `simulationStore.viewState` 保存转换后的前端视图数据。

数据转换在 `backendSnapshotAdapter.js`：

- `backendSnapshotAdapter.js:46-61` 的 `applyCellToGrid(...)`：把后端 map cell 转成前端 grid node 状态。
- `backendSnapshotAdapter.js:63-76` 的 `adaptVehicle(...)`：把后端车辆位置 `x/y` 转成前端 `row/col`。
- `backendSnapshotAdapter.js:78-87` 的 `adaptFrontier(...)`：转换 frontier。
- `backendSnapshotAdapter.js:114-135` 的 `remainingPathForTask(...)`：根据任务 `pathQueue` 生成剩余路径。
- `backendSnapshotAdapter.js:137-190` 的 `adaptSnapshot(...)`：汇总生成完整 viewState。

这些数据会传给 `VisualizerCanvas.vue`。传参位置是 `PathfindingVisualizer.vue:3-31`：

- `PathfindingVisualizer.vue:19` 传入 `vehicles`。
- `PathfindingVisualizer.vue:21` 传入 `frontiers`。
- `PathfindingVisualizer.vue:22` 传入 `pathsByVehicle`。
- `PathfindingVisualizer.vue:27` 传入 `backendChangedCells`。

`VisualizerCanvas.vue` 监听这些数据变化的位置是：

- `VisualizerCanvas.vue:168-172` 的 `backendChangedCells` watcher。
- `VisualizerCanvas.vue:237-241` 的 `vehicles` watcher。
- `VisualizerCanvas.vue:243-247` 的 `frontiers` watcher。
- `VisualizerCanvas.vue:249-253` 的 `pathsByVehicle` watcher。

实际刷新方法：

- `VisualizerCanvas.vue:606-614` 的 `updateChangedGridCells(...)`：更新变化的地图格子。
- `VisualizerCanvas.vue:685-710` 的 `updateVehicleMeshes()`：更新小车模型位置和朝向。
- `VisualizerCanvas.vue:843-885` 的 `updateFrontierMeshes()`：更新 frontier 标记。
- `VisualizerCanvas.vue:887-924` 的 `updatePathMeshes()`：更新路径线。
- `VisualizerCanvas.vue:1304-1325` 的 `updateNode(...)`：根据节点状态渲染墙、访问区、未知区等。

最后，`VisualizerCanvas.vue:472-486` 的 `renderLoop()` 持续渲染 Three.js 场景，所以前端看到的小车移动，本质上是后端状态变化后的实时可视化。

## 十二、完整讲解版

给老师讲的时候，可以按下面这段说：

```text
小车探索是前后端分工完成的。前端主要负责用户操作和 3D 展示，后端负责真正的探索逻辑。

页面打开后，PathfindingVisualizer.vue:474-486 的 initBackend() 会先通过 api.js:86-88 的 fetchState() 请求 main.py:331-336 的 get_state(...) 接口，获取当前地图、小车、任务和路径状态。同时，ws.js:4-53 的 createSimulationSocket(...) 会连接 main.py:647-657 的 websocket_endpoint(...)，等待后端推送实时快照。

所以整个流程可以概括为：前端发启动命令，后端执行探索逻辑，WebSocket 实时推送状态，前端把状态可视化。
```
