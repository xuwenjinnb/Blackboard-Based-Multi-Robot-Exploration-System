# 软件设计阶段成果演示讲稿

## 1. 开场

老师好，我这次展示的是“多小车协作巡检仿真系统”在软件设计阶段完成的任务。这个阶段我主要做的是把前面的需求，转换成可以指导后续编码和测试的设计文档。展示重点包括总体架构、数据结构、详细设计、接口设计和模块协作流程。

## 2. 完成任务总览

本阶段完成了三类工作：

- 总体架构设计：确定前后端分离、FastAPI 后端、Vue/Three.js 前端、Redis 黑板和三类 Worker 的协作方式。
- 数据与模块设计：整理车辆、地图、任务、前沿点、导航请求、路径规划、心跳和事件等数据结构，并细化小车运行流程。
- 接口与协作设计：整理 HTTP REST、WebSocket、Redis 黑板接口，并用顺序图说明一次仿真运行中的组件调用关系。

## 3. 总体架构设计

系统采用前后端分离和 Redis 黑板架构。前端只访问 FastAPI 后端，不直接访问 Redis。后端负责认证、配置、控制、状态快照和回放查询。Robot Worker、Controller Worker、Navigator Worker 之间不直接互相调用，而是都读写 Redis 黑板。

这样设计的好处是模块解耦：Controller 只负责任务分配，Navigator 只负责路径规划，Robot 只负责扫描和执行。后续如果替换任务分配策略或路径规划算法，不需要重写整个系统。

## 4. 功能边界与角色设计

系统设计了三类用户：

- 管理员：负责系统运行员和分析员账号管理。
- 系统运行员：负责地图、小车、算法配置，并启动、暂停、继续、重置仿真。
- 分析员：负责查看历史回放和运行分析结果。

角色边界明确后，后续接口权限、页面入口、数据访问范围就都有了设计依据。

## 5. 运行状态设计

小车状态设计是详细设计的基础。小车从待分配状态开始，控制器写入任务后进入分配中；导航器生成有效路径后进入运行中；运行中小车按路径移动并更新地图；如果任务完成就回到待分配，如果发生碰撞风险或阻塞，就回退并触发重规划。

这套状态设计保证了小车模块、任务模块和前端显示使用同一套状态语义。

## 6. 数据结构设计

系统运行时的数据主要放在 Redis 黑板中。核心对象包括：

- RuntimeConfigInfo：运行策略、导航算法、导航器数量。
- VehicleInfo：小车位置、朝向、电量、状态、当前任务。
- FrontierInfo：前沿点位置、未知收益、分配状态。
- TaskInfo：目标点、路径队列、执行进度、重规划次数。
- NavigationRequestInfo 和 NavigationPlanInfo：路径规划请求和结果。
- MapInfo 和 MapCellInfo：地图尺寸、版本、格子状态。
- ComponentHeartbeatInfo 和 EventLogInfo：组件心跳和系统事件。

通过这些结构，Controller、Navigator、Robot 可以围绕统一黑板交换状态。

## 7. 地图探索主流程

地图探索流程被设计成一个闭环：

扫描地图 -> 发现 frontier 前沿点 -> 控制器分配任务 -> 导航器规划路径 -> 小车执行移动 -> 更新地图和车辆状态 -> 前端刷新显示。

这一部分的数据流图明确了每个处理过程读写哪些数据表，例如车辆状态表、任务路径队列、地图视图表、事件表和规划时间表。

## 8. 小车模块详细设计

小车模块进一步分成二层和三层设计。

二层图说明小车收到节拍后，先读取自身状态，再根据待分配、分配中、运行中三种状态走不同处理流程。

三层图展开“运行中状态”：小车读取路径队列，判断路径是否存在，取下一路径点，进行碰撞风险检测，执行移动，点亮周围地图，更新任务进度。如果路径结束则完成任务，如果阻塞则回退并写入重规划事件。

## 9. 接口设计

接口分为三类：

- HTTP REST 接口：用于登录认证、提交配置、启动/暂停/重置仿真、回放查询。
- WebSocket 接口：用于后端持续向前端推送地图、小车、任务、路径和事件状态。
- Redis 黑板接口：用于 Controller、Navigator、Robot 读写任务、路径、车辆、地图、事件和心跳。

设计原则是：前端只和 FastAPI 后端通信；内部 Worker 不额外暴露 HTTP 服务，只通过 Redis 黑板协作。

## 10. 顺序协作设计

一次完整仿真运行流程是：

运行员在 Dashboard 提交配置，后端写入 runtime、地图和车辆初始状态；控制器读取空闲小车和前沿点，创建任务和导航请求；导航器领取导航请求并写回路径计划；小车读取路径执行移动并上传地图 patch；后端从 Redis 读取状态快照，通过 WebSocket 推送给前端。

顺序图把静态结构和动态运行流程连接起来，说明整个设计是可以闭环执行的。
