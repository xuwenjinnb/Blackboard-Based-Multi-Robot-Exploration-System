# 多泳道系统流程图说明

对应图片：

```text
fig/系统流程图/多泳道系统流程图.svg
```

新版按参考图格式绘制，但重新压缩了信息密度，并把 Redis 黑板拆成独立泳道。图中只展示“登录与三角色分流”和“系统运行员启动后的黑板协作主线”，不展开 token、接口地址、算法公式和 Redis key 细节，避免流程线过多。

## 1. 泳道划分

从左到右共 6 个泳道：

| 泳道 | 表达内容 |
| --- | --- |
| 界面 | 登录、三角色分流、运行员配置和实时展示 |
| 后端服务 | 接收界面请求、从 Redis 读取用户信息并认证、处理用户管理/回放/运行配置请求、生成状态快照 |
| Redis 黑板 | 只作为后端和组件访问的共享状态中心，保存用户会话、回放数据、运行配置、地图、车辆、任务、导航请求和路径计划 |
| 小车 | 读取任务、扫描地图、执行路径、更新车辆状态 |
| 控制器 | 读取 frontier 和空闲小车，创建任务和导航请求 |
| 导航器 | 领取导航请求，规划路径，写回路径计划 |

## 2. 主流程

图的阅读顺序是：

```text
开始
  -> 显示登录界面
  -> 输入账号密码
  -> 后端从 Redis 读取用户信息并完成登录认证
  -> 返回登录结果和角色
  -> 判断角色
  -> 管理员进入用户管理
  -> 分析员进入回放分析
  -> 系统运行员进入 Dashboard
      -> 配置地图、障碍、小车、任务策略和路径算法
      -> 调用后端服务提交配置
      -> 后端把 runtime config 写入 Redis 黑板
      -> 点击开始探索
      -> 后端把 systemStatus=RUNNING 写入 Redis 黑板
      -> 小车、控制器、导航器只通过 Redis 黑板读写状态
      -> 前端实时显示地图、车辆、任务、路径和事件
```

## 3. 角色分支

登录成功后有三个角色分支：

| 角色 | 功能 |
| --- | --- |
| 管理员 admin | 用户管理：新增、编辑、删除系统运行员和分析员 |
| 分析员 analyst | 回放分析：查看历史记录、播放回放、查看黑板数据 |
| 系统运行员 operator | 配置并启动多小车巡检仿真，是下半区主流程的触发者 |

图中必须体现三个角色，但管理员和分析员只画成短支线，不进入 Robot、Controller、Navigator 主流程；系统运行员才进入下半区的多车巡检闭环。管理员和分析员也不能直接访问 Redis 黑板，它们只通过界面向后端服务发送请求，再由后端服务读写 Redis。

## 4. 黑板协作逻辑

系统运行员点击开始探索后，核心循环是：

```text
Robot 小车
  -> 读取任务和 pathQueue
  -> 没有路径时扫描地图、发现 frontier
  -> 有路径时执行 pathQueue、更新车辆状态
  -> 写回 Redis 黑板

Controller 控制器
  -> 读取 OPEN frontier 和 IDLE 小车
  -> 判断是否有可分配目标
  -> 创建 task 和 navigationRequest
  -> 写回 Redis 黑板

Navigator 导航器
  -> 读取 PENDING request 和地图
  -> 规划路径
  -> 写入 SUCCESS / NO_PATH
  -> 更新 task.pathQueue
```

Redis 黑板保存：

```text
map
vehicles
frontiers
tasks
navigationRequests
navigationPlans
heartbeats
events
replays
```

## 5. 图中为什么不写 token

token 是登录成功后后端发给前端的临时身份凭证。登录时，后端会先从 Redis 中读取用户信息，校验账号、密码哈希和账号状态；登录成功后再生成 token，让后端在后续请求中识别“当前请求是谁发的、是什么角色”。代码中仍然保留 token，用于接口权限和 WebSocket 连接校验。

但是流程图里不建议单独写“生成 token”，因为老师看系统流程时更关心“是否完成登录认证、是否能按角色分流”。所以图中统一写成：

```text
登录认证
读取用户信息
返回登录结果和角色
```

## 6. 答辩讲解词

```text
这张多泳道图分成上下两部分。上半部分体现登录和角色分流：管理员进入用户管理，分析员进入回放分析，系统运行员进入 Dashboard。管理员和分析员是权限分支，不参与实时小车调度；真正触发多车巡检主流程的是系统运行员。

下半部分展示系统运行员点击开始探索后的 Redis 黑板协作闭环。系统运行员进入 Dashboard 后填写地图、障碍、小车数量、任务分配策略和路径规划算法，界面先调用后端服务把 runtime config 写入 Redis 黑板；点击开始探索后，界面再次调用后端服务，由后端把 systemStatus=RUNNING 写入 Redis 黑板。Redis 黑板作为独立共享状态中心，保存 map、vehicles、frontiers、tasks、navigationRequests、navigationPlans、heartbeats、events 和 replays。小车、控制器、导航器之间不直接调用，而是分别读写 Redis 黑板。前端通过 WebSocket 接收后端生成的状态快照并实时显示运行过程。
```
