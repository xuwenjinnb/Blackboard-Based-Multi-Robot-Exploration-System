# 数据描述及三个需求

本文档整理“多小车协作巡检仿真系统”的数据描述、性能需求、运行需求和其它需求。内容以当前项目实现为依据：前端采用 Vue/Three.js 展示地图和小车状态，后端采用 FastAPI 提供 REST 接口与 WebSocket 快照推送，Redis 黑板保存用户、地图、车辆、任务、导航、事件和回放数据。

## 2.3 数据描述

### 2.3.1 数据约束定义

| 序号 | 数据项 | 形式化描述 | 项目对应说明 |
| --- | --- | --- | --- |
| 1 | 账户 | `User = username + password + role` | 用户数据保存在 Redis 用户表中，登录后生成 Bearer Token 会话。 |
| 1.1 | 用户名 | `username = 4{字符}8` | 报告约束为 4 到 8 个字符；当前代码允许字母、数字、点、下划线、短横线，长度为 1 到 40。 |
| 1.2 | 密码 | `password = 6{字符}12` | 报告约束为 6 到 12 个字符；当前代码要求不少于 6 位，并以加盐哈希保存，不保存明文密码。 |
| 1.3 | 角色 | `role = [Configurator \| Analyst \| Admin]` | 当前项目实际角色为 `operator`、`analyst`、`admin`。其中 `operator` 对应配置与运行控制角色，即 Configurator/系统运行员。 |
| 2 | 算法 | `Algorithm = [0 \| 1 \| 2]` | 对应任务分配策略：`0=nearest` 最近可达前沿，`1=greedy` 信息增益优先，`2=low_mdp` 低层 MDP 探索。导航器算法另有 `baseline` 与 `cbs`。 |
| 3 | 地图规模 | `mapSize = 1..100` | 报告约束为 1 到 100；当前实现用 `width`、`height` 表示二维网格，默认 `32x24`，代码上限为 `512x512`，验收场景建议控制在 `100x100` 以内。 |
| 4 | 地图探索区域 | `MapView = 1{isExplored}100`，`isExplored = [0 \| 1]` | 当前实现中地图由 `map.cells` 保存，单元状态包括 `UNKNOWN`、`FREE`、`VISITED`、`OBSTACLE`。当状态不是 `UNKNOWN` 时，`isExplored=1`。 |
| 5 | 障碍探索区域 | `BlockView = 1{isBlocked}100`，`isBlocked = [0 \| 1]` | 当前实现中真实障碍保存在 `true_obstacles`，感知地图中障碍状态为 `OBSTACLE`。当单元格为障碍时，`isBlocked=1`。 |
| 6 | 回放日志总文件数 | `file_num = 0..n` | 对应回放索引数量，即 `/replays` 返回的回放记录数。 |
| 7 | 回放日志号 | `order_file_num = 0..n` | 对应用户选择的回放序号或 `replayId`，其中 `n` 与 `file_num` 的上限一致。 |
| 8 | 记录总轮数 | `last_view = 0..m` | 对应某个回放中的状态帧总数，即回放元数据中的 `frameCount`。 |
| 9 | 回放开始轮数 | `order_view = 0..m` | 对应回放播放器选定的起始帧或当前播放帧，其中 `m` 与 `last_view` 的上限一致。 |
| 10 | 导航器状态 | `navigator_status = 4{navStatus}4`，`navStatus = [false \| true]` | 当前项目用 `heartbeats` 保存导航器心跳，`READY` 或 `BUSY` 可视为 `true`，无心跳或异常可视为 `false`。如需固定 4 个导航器，可将 `navigatorCount` 配置为 4。 |

### 2.3.2 项目核心数据对象

| 数据对象 | 关键字段 | 说明 |
| --- | --- | --- |
| 用户 `User` | `username`、`displayName`、`role`、`enabled`、`salt`、`passwordHash` | 支持管理员、系统运行员和分析员三类用户，密码以加盐哈希形式存储。 |
| 登录会话 `Session` | `token`、`username`、`createdAt`、`TTL` | 登录成功后创建，REST 接口与 WebSocket 都通过 token 鉴权。 |
| 地图 `Map` | `mapId`、`width`、`height`、`chunkSize`、`version`、`generation`、`cells` | 地图按网格保存，并支持 `mapDelta` 增量推送，降低前端刷新开销。 |
| 地图单元 `Cell` | `x`、`y`、`state`、`confidence`、`updatedBy`、`updatedAt` | `state` 表示未知、空闲、已访问或障碍，坐标精度为 1 个网格。 |
| 小车 `Vehicle` | `vehicleId`、`pose.position`、`pose.heading`、`status`、`battery`、`currentTaskId` | 表示小车实时位置、朝向、任务和电量状态。 |
| 前沿点 `Frontier` | `frontierId`、`position`、`unknownGain`、`status` | Robot 扫描后发现，Controller 根据策略将其分配给空闲小车。 |
| 任务 `Task` | `taskId`、`vehicleId`、`target`、`status`、`pathQueue`、`currentStepIndex` | 表示某辆小车前往某个目标点的探索任务。 |
| 导航请求 `NavigationRequest` | `requestId`、`vehicleId`、`taskId`、`start`、`goal`、`priority`、`status` | Controller 创建，Navigator 领取后进行路径规划。 |
| 路径计划 `NavigationPlan` | `planId`、`requestId`、`taskId`、`vehicleId`、`path`、`planner`、`status` | Navigator 写回 Redis，Robot 按 `pathQueue` 执行移动。 |
| 心跳 `Heartbeat` | `componentId`、`componentType`、`status`、`currentWorkId`、`updatedAt` | 记录 Robot、Controller、Navigator 等组件在线和工作状态。 |
| 事件 `Event` | `eventId`、`source`、`eventType`、`message`、`timestamp` | 用于运行过程追踪、状态面板展示和问题排查。 |
| 回放 `Replay` | `replayId`、`operator`、`status`、`frameCount`、`frames` | 系统启动探索后记录状态帧，分析员可加载回放进行检查。 |

## 2.4 性能需求

### 2.4.1 数据精确度

| 指标 | 需求 | 项目对应措施 |
| --- | --- | --- |
| 地图探索精度 | 地图探索数据精度达到 99.9% 以上，确保无遗漏区域。 | Robot 每次扫描后上传地图 patch，Redis 黑板按 `map.version` 合并，前端按 `mapDelta` 同步。 |
| 路径规划误差 | 实际路径与理论最优路径偏差不超过 10%。 | `baseline` 使用 A*/时空 A*，优先保证可达性；`cbs` 用于多车冲突规避。 |
| 小车位置精度 | 小车位置坐标精度为 1 个网格单位。 | 小车位置统一使用 `{x, y}` 网格坐标，前端按网格位置渲染车辆模型。 |
| 障碍检测误报率 | 障碍物检测误报率小于 0.1%。 | 障碍以 `true_obstacles` 和 `OBSTACLE` 状态记录，扫描结果写入感知地图。 |
| 小车碰撞率 | 小车碰撞率小于 0.1%。 | 路径规划时考虑障碍和车辆占用位置，时空 A* 和 CBS 用于减少多车路径冲突。 |

### 2.4.2 时间特性

| 指标 | 需求 | 项目对应说明 |
| --- | --- | --- |
| 路径规划计算时间 | 在 `100x100` 网格规模下，单次路径规划时间不超过 10ms。 | 对普通单车请求优先使用 A*/时空 A*；CBS 批量规划复杂度更高，必要时可限制批大小或退回 baseline。 |
| 小车动作延迟 | 从接收指令到开始移动不超过 200ms。 | 分布式模式下可通过 `WORKER_INTERVAL_SECONDS`、`robot_worker --steps-per-tick` 调整动作推进频率。 |
| 界面刷新频率 | 前端刷新频率不低于 10Hz。 | WebSocket 推送 `STATE_SNAPSHOT`，性能模式可将 `BROADCAST_INTERVAL_SECONDS` 配置到 0.1s 左右。 |
| Redis 同步时效 | Redis 数据写入到读取延迟不超过 50ms。 | Redis 黑板采用 Hash、Set、Stream、List 保存状态，并使用 pipeline/lock 保证一致读写。 |
| 消息端到端时延 | RabbitMQ 消息端到端传递时间不超过 30ms。 | 当前项目以 Redis 黑板和 WebSocket 为主，RabbitMQ 可作为后续分布式消息队列扩展指标。 |

### 2.4.3 适应性

1. 系统应适应不同障碍物密度，支持手动障碍、随机障碍和按密度生成障碍三种配置方式。
2. 前端界面应根据地图规模自动调整方格大小和三维视角，保证小地图清晰、大地图可浏览。
3. 导航器应根据地图大小、车辆数量和任务数量调整规划方式。当前项目支持 `baseline` 与 `cbs`，并允许配置 `navigatorCount`。
4. 小车应根据地图规模和运行节奏动态调整等待导航器分配路径的时间，避免在大地图或多车场景中频繁空转。
5. 地图数据应支持分块和增量同步，降低大地图下完整快照传输的开销。

## 2.5 运行需求

### 2.5.1 用户界面

1. 交互规范应符合 GUI 设计三原则：一致性、反馈性和简洁性。
2. 登录后按角色进入不同界面：管理员进入用户管理，系统运行员进入仿真控制台，分析员进入回放分析界面。
3. 关键操作应提供明确反馈。启动、暂停、继续、停止、重置、删除回放、删除用户等操作应在界面上显示成功或失败信息。
4. 对终止探索、重置系统、删除用户、删除回放等不可直接撤销的操作，应提供确认对话框。
5. 可视化应使用颜色或模型区分状态：已探索区域、未知区域、障碍物、小车实时位置、前沿点和规划路径。
6. 小车应显示实时位置和朝向。当前前端使用 Three.js 和车辆模型展示不同方向的小车状态。

### 2.5.2 硬件接口

1. 系统应支持鼠标输入，包括按钮点击、地图拖动、障碍编辑、视角缩放和回放进度控制。
2. 系统可支持键盘作为辅助输入，例如表单输入、快捷控制和浏览器默认操作。

### 2.5.3 软件接口

| 接口类型 | 接口内容 | 当前项目对应 |
| --- | --- | --- |
| 认证接口 | 登录、登出、当前用户查询 | `/auth/login`、`/auth/logout`、`/auth/me` |
| 用户管理接口 | 用户列表、新增、修改、删除 | `/admin/users` |
| 运行配置接口 | 地图、障碍、小车、策略、导航器配置 | `/runtime/map`、`/runtime/obstacles`、`/runtime/vehicles`、`/runtime/policy`、`/runtime/navigator` |
| 控制接口 | 启动、暂停、继续、停止、重置 | `/control/start`、`/control/pause`、`/control/resume`、`/control/stop`、`/control/reset` |
| 状态接口 | 获取当前状态和运行参数 | `/state`、`/runtime` |
| 实时推送接口 | 状态快照、地图增量、车辆状态、任务和事件 | `/ws` WebSocket |
| 回放接口 | 回放列表、回放加载、回放删除 | `/replays`、`/replays/{replayId}` |
| Redis 接口 | 共享黑板数据读写 | 当前使用 Python `redis` 客户端，保存 `map`、`vehicles`、`tasks`、`navigationRequests`、`navigationPlans`、`heartbeats`、`events`、`replays` 等数据。 |
| RabbitMQ 接口 | 异步消息队列通信 | 当前项目未接入 RabbitMQ，可作为后续扩展，按 AMQP 0-9-1 协议实现组件间消息传递。 |

小车数据、导航请求、路径计划和地图补丁均采用 JSON 格式传输。例如控制器通知导航器分配路径时，导航请求包含小车 ID、起点、目标点和任务 ID：

```json
{
  "requestId": "request-1",
  "vehicleId": "car-01",
  "taskId": "task-1",
  "start": { "x": 3, "y": 5 },
  "goal": { "x": 20, "y": 12 },
  "priority": 5,
  "status": "PENDING"
}
```

### 2.5.4 故障处理

1. Redis 断连时，组件应进行重试。当前 worker 已具备 Redis 不可用时等待并重试的基础机制；需求目标可完善为指数退避策略，最大重试 5 次。
2. Redis 作为共享黑板保存全局状态。当某个组件断联或出故障时，车辆、地图、任务、路径、心跳和事件数据仍保存在 Redis 中。
3. 组件重新启动后，应根据 Redis 中的当前全局状态继续运行，避免因为单个组件重启导致整套系统状态丢失。
4. 对不可恢复的系统错误，应保证数据不被破坏，并尽量保留 Redis 当前状态、回放帧和事件日志，便于恢复和排查。
5. 探索完成、停止或重置时，应结束当前回放记录，保存关键状态帧，供分析员后续检查小车运动轨迹和系统事件。

## 2.6 其它需求

### 2.6.1 安全性

1. 用户密码必须加密或哈希存储，不能保存明文密码。当前项目采用 PBKDF2-HMAC-SHA256 加 salt 的方式；若严格按需求文档，可升级为 BCrypt 哈希加 salt。
2. 系统应使用基于角色的权限控制。管理员负责用户管理，系统运行员负责配置和控制仿真，分析员负责查看回放。
3. 登录会话应设置有效期，避免 token 长期有效。当前项目会话 TTL 为 12 小时。
4. WebSocket 连接也需要 token 鉴权，防止未登录用户直接订阅系统状态。
5. 如后续引入 RabbitMQ，应启用虚拟主机、账户和密码认证，并限制不同组件的访问权限。
6. Redis 在部署时应限制外部访问，生产环境建议启用密码、内网访问或容器网络隔离。

### 2.6.2 可维护性

1. 系统采用模块化设计，前端、API 网关、Redis 黑板、Controller、Navigator 和 Robot 组件之间保持低耦合。
2. 各组件之间通过 Redis 黑板交换状态，不直接依赖彼此进程地址，因此可以独立启动、独立部署和独立恢复。
3. 当前项目已经提供独立 worker 和 exe 入口，包括 `api_gateway`、`controller_worker`、`navigator_worker`、`robot_worker` 和 `display_ui`。
4. 日志和事件应覆盖关键流程，包括系统控制、地图更新、任务分配、路径规划成功/失败、小车阻塞和探索完成。
5. 探索完成后，系统应保留回放日志，便于检查小车运动是否异常、路径是否冲突、障碍是否识别正确。
6. 后端测试应覆盖 Redis 黑板持久化、路径规划、认证权限和回放读写等关键功能。

### 2.6.3 未来扩展

1. 预留 ROS 接口，支持后续接入实体机器人，将仿真小车状态映射为真实机器人话题或服务。
2. 支持导入和导出地图配置文件，建议使用 YAML 格式保存地图规模、障碍物、小车初始位置和算法配置。
3. 支持动态扩展地图规模，在大地图下继续使用分块存储和 `mapDelta` 增量同步。
4. 支持更多路径规划算法和任务分配策略，例如 D* Lite、强化学习策略、多目标优化和能耗约束规划。
5. 支持 RabbitMQ 等消息队列作为 Redis 黑板之外的异步事件通道，用于更大规模的分布式部署。
6. 支持多显示端同时观察系统状态，便于课堂演示、调试和远程验收。

## 与当前项目实现的对应说明

当前项目的核心实现更偏向“Redis 黑板 + FastAPI + WebSocket + 分布式 worker”。因此，文档中 RabbitMQ、Jedis、BCrypt 等内容应理解为需求目标或未来扩展项；当前代码中实际采用的是 Python `redis` 客户端、Redis 数据结构、WebSocket 状态推送，以及 PBKDF2-HMAC-SHA256 加盐哈希。若后续需要严格满足原始需求，可在现有架构上补充 RabbitMQ 消息队列、BCrypt 密码哈希和 Redis 指数退避重试策略。
