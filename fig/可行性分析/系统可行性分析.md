# 多小车协作巡检仿真系统可行性分析

## 1.1 项目简介

### 1.1.1 项目目标

本项目拟开发一个“多小车协作巡检仿真系统”，用于模拟多辆小车在未知或半未知地图环境中的协同探索、巡检覆盖、任务分配和路径规划过程。系统面向课程实验、软件构造设计展示和多车协作算法验证，目标不是简单展示小车动画，而是形成从环境配置、车辆部署、感知探索、任务调度、路径规划、执行反馈到数据回放的完整软件闭环。

1）实现多小车协作巡检的核心运行闭环。小车能够在地图中移动并扫描周围环境，系统能够根据扫描结果更新地图状态、发现 frontier 前沿点、生成巡检任务、分配空闲车辆、规划路径并驱动车辆继续探索，最终提高地图覆盖率并完成巡检过程展示。

2）实现可配置、可切换的仿真环境。系统运行员可以在前端配置地图大小、障碍物、小车数量、导航器数量、任务分配策略和路径规划算法，使系统能够适应不同规模、不同障碍密度和不同算法组合下的巡检仿真需求。

3）实现低耦合、可扩展的软件结构。系统采用 FastAPI 后端、Redis 黑板、Robot 组件、Controller 组件、Navigator 组件和 Vue/Three.js 前端分层设计。Robot、Controller、Navigator 之间不直接相互调用，而是通过 Redis 黑板读写共享状态，从而降低模块耦合度，并支撑当前已经完成的分布式 worker 部署和后续算法扩展。

4）实现直观的三维可视化和实时状态反馈。前端通过 WebSocket 接收后端状态快照，使用 Three.js 展示地图、障碍、小车模型、frontier、规划路径和运行状态，同时展示覆盖率、步数、任务队列、导航请求、路径计划和系统事件，使用户能够直观看到多车协作巡检过程。

5）实现用户分级管理和运行记录保存。系统设置超级管理员、系统运行员和分析员三类角色，分别承担用户管理、仿真配置与控制、历史回放分析等职责。系统在仿真运行过程中保存回放数据，便于后续查看小车轨迹、任务变化、路径规划结果和事件日志。

### 1.1.2 系统的简要描述

本系统是一个基于浏览器的多小车协作巡检仿真平台。前端采用 Vue 和 Three.js 构建交互界面，用户登录后会根据角色进入不同功能页面：超级管理员进入用户管理界面，系统运行员进入巡检仿真 Dashboard，分析员进入历史回放分析界面。系统运行员可以在 Dashboard 中完成地图、障碍、小车和算法配置，并启动、暂停、继续、停止或重置巡检仿真。

后端采用 Python 和 FastAPI 实现，负责登录认证、角色权限控制、运行配置、仿真控制、状态查询、WebSocket 推送和回放查询等功能。系统提供 `/auth/login`、`/admin/users`、`/runtime/*`、`/control/*`、`/state`、`/ws`、`/replays` 等接口，前端通过 REST 接口发送配置和控制命令，通过 WebSocket 接收实时状态快照。

系统的数据中心为 Redis 黑板。用户、登录会话、地图、车辆、frontier、任务、导航请求、路径计划、组件心跳、系统事件和回放数据均保存在 Redis 中。仿真运行时，Robot 负责小车感知与执行，Controller 负责任务分配，Navigator 负责路径规划。三类组件围绕 Redis 黑板交换状态，形成“扫描地图 -> 发现前沿点 -> 分配任务 -> 规划路径 -> 执行移动 -> 更新地图”的循环。

当前系统既支持单机内嵌仿真演示，也支持分布式 worker 部署。单机模式下，FastAPI 后端内部调度 SimulationEngine 完成完整流程；分布式模式下，FastAPI 和 Redis 可以部署在中心计算机，Controller Worker、Navigator Worker 和 Robot Fleet Worker 可以部署在另一台计算机，通过同一个 Redis 黑板协作运行。

## 1.2 对现有系统的分析

现有系统即当前项目已经实现的“多小车协作巡检仿真平台”。从项目文件和代码结构看，系统已经具备较完整的前后端功能、角色权限体系、Redis 黑板数据模型、路径规划算法、实时可视化界面和回放分析能力。其主要功能如下。

### 1.2.1 主要功能

1）登录认证与角色分级功能

系统支持基于账号密码的登录认证，登录成功后生成 token 会话，并根据角色进入不同页面。超级管理员负责管理系统运行员和分析员账号，可以新增、编辑、删除、启用或停用用户；系统运行员负责仿真配置和巡检运行控制；分析员负责查看历史回放和运行数据。角色分级使不同用户只访问与自身职责相关的功能，降低误操作风险。

2）仿真环境配置功能

系统运行员可以在前端配置地图宽度、高度、分块大小、障碍物数量或障碍物单元、小车数量、导航器数量、任务分配策略和路径规划算法。系统支持随机部署障碍物和车辆，也支持对障碍单元进行局部配置。为保证运行一致性，后端在仿真 RUNNING 或 PAUSED 状态下会限制地图、障碍和车辆数量等关键配置的修改。

3）巡检运行控制功能

系统提供启动、暂停、继续、停止和重置等运行控制能力，对应后端 `/control/start`、`/control/pause`、`/control/resume`、`/control/stop` 和 `/control/reset` 接口。运行状态会写入 Redis 黑板，前端再通过状态快照同步显示。每次启动仿真时，系统会创建回放记录；停止或重置时，系统会结束并保存当前回放。

4）多小车协作探索功能

Robot 组件负责小车的感知和执行，包括扫描周围环境、上传地图 patch、更新车辆状态、发现 frontier 前沿点、执行路径队列以及在遇到阻塞时触发重规划。Controller 组件读取 Redis 中的开放 frontier 和空闲车辆，根据任务分配策略生成任务和导航请求。Navigator 组件领取导航请求后规划路径，并将路径计划写回黑板，小车再按路径队列移动，从而形成多车协作探索闭环。

5）任务分配与决策策略功能

系统支持多种任务分配和探索策略。`nearest` 策略优先将空闲小车分配给距离最近且可达的 frontier，适合稳定演示；`greedy` 策略综合考虑 frontier 附近未知区域收益和路径代价，更强调探索收益；`low_mdp` 策略让小车跳过全局任务分配和全局导航规划，直接基于局部价值迭代进行自主探索，体现更去中心化的决策方式。

6）路径规划与冲突处理功能

系统提供 `baseline` 和 `cbs` 等导航算法配置。基础导航模式主要使用 A* 或时空 A* 完成单车路径规划，并考虑障碍与路径可达性；CBS 冲突搜索适合多车联合路径规划，可用于处理多辆车在同一时间占用同一网格或相向交换位置等冲突问题。路径规划结果以 NavigationPlan 的形式写入 Redis，供 Robot 执行。

7）实时三维可视化功能

前端 Dashboard 使用 Three.js 展示三维地图、小车模型、障碍物、已探索区域、frontier 前沿点和规划路径。系统通过 WebSocket 持续推送状态快照，并支持 `mapDelta` 地图增量同步，使前端能够实时刷新车辆位置、覆盖率、步数、任务队列、导航请求、路径计划、组件心跳和事件日志。

8）Redis 黑板共享状态功能

Redis 黑板是系统的共享状态中心，保存地图、车辆、frontier、任务、导航请求、路径计划、心跳、事件和回放等数据。FastAPI、Robot、Controller、Navigator 和前端显示逻辑都围绕 Redis 状态进行协作。该结构使组件之间不依赖彼此的进程地址和直接函数调用，有利于独立开发、独立部署和故障恢复。

9）历史回放与数据分析功能

系统在每次仿真运行过程中记录状态帧，保存地图、车辆、覆盖率、任务、导航和事件等数据。分析员可以查看回放列表、加载单次回放、播放或暂停回放、查看对应数据明细，必要时也可以删除回放记录。该功能有助于复盘小车运动轨迹、任务分配效果、路径规划结果和异常事件。

10）分布式部署与可执行文件支持功能

当前项目不仅保留单机演示能力，还提供分布式运行脚本和组件入口。系统可以将 FastAPI、前端静态页面和 Redis 黑板部署在一台计算机上，将 Controller Worker、Navigator Worker 和 Robot Fleet Worker 部署在另一台计算机上。项目中也包含 `api_gateway`、`controller_worker`、`navigator_worker`、`robot_worker` 和 `display_ui` 等可执行入口，便于验收演示和后续部署。

### 1.2.2 现有系统运行流程

现有系统的主要运行流程为：用户先登录系统，后端根据角色加载相应界面；系统运行员进入 Dashboard 后配置地图、障碍、小车数量、任务分配策略和路径规划算法；点击启动后，FastAPI 将运行状态和配置写入 Redis 黑板；Robot 扫描环境并上传地图更新，同时发现 frontier；Controller 读取 frontier 和空闲车辆并生成任务与导航请求；Navigator 领取导航请求并规划路径；Robot 读取路径计划并执行移动；前端通过 WebSocket 持续接收状态快照并刷新三维界面；系统停止或重置后保存回放数据，供分析员后续查看。

### 1.2.3 现有系统特点

现有系统的特点可以概括为：功能链路较完整，已经覆盖登录、配置、运行、可视化、回放和用户管理等主要业务；系统架构较清晰，前端、后端、Redis 黑板、Robot、Controller 和 Navigator 职责分明；算法具有可切换性，任务分配策略和导航规划算法均可在配置中选择；可视化效果较直观，能够实时呈现多小车探索过程；扩展基础较好，Redis 黑板和 worker 设计已经支撑分布式部署，并为新算法接入和真实小车控制预留了空间。

## 1.3 技术可行性分析

1）技术架构选择：

本系统采用“前端展示层 + FastAPI 接口层 + Redis 黑板层 + 分布式 Worker 组件层”的总体架构。前端负责用户交互和三维可视化展示，FastAPI 负责登录认证、运行配置、状态接口、WebSocket 推送和前端静态资源服务，Redis 作为共享状态中心保存地图、车辆、任务、导航、事件和回放数据，Controller Worker、Navigator Worker 和 Robot Fleet Worker 分别负责任务分配、路径规划和车辆执行。

后端使用 Python 语言和 FastAPI 框架实现。Python 生态成熟，适合快速实现路径规划、仿真调度和多车协作探索等算法逻辑；FastAPI 具有开发效率高、接口结构清晰、异步支持较完善等优点，能够满足系统实时交互、状态查询、权限控制和 WebSocket 推送需求。当前项目不仅具备单机内嵌仿真能力，也已经完成了分布式低耦合设计：`run_api_distributed.ps1` 可启动仅承担接口网关和状态广播职责的 FastAPI 服务，`run_controller_worker.ps1`、`run_navigator_worker.ps1`、`run_robot_worker.ps1` 可分别启动独立 worker 进程，系统还提供 `api_gateway.exe`、`controller_worker.exe`、`navigator_worker.exe`、`robot_worker.exe` 和 `display_ui.exe` 等可执行组件。

系统使用 Redis 作为黑板状态中心。地图、车辆、frontier 边界点、任务、导航请求、路径规划结果、组件心跳、事件日志和回放数据均保存在 Redis 中。FastAPI、Controller、Navigator 和 Robot 之间不依赖直接函数调用或彼此进程地址，而是通过 Redis 读写共享状态完成协作。这种设计已经能够支持“两台计算机分布式部署”：一台计算机运行 FastAPI、前端静态页面和 Redis，另一台计算机运行 Controller Worker、Navigator Worker 和 Robot Fleet Worker，各组件通过同一个 Redis 黑板交换数据。

前端采用 Vue 和 Three.js 等技术实现三维可视化界面。系统能够在浏览器中展示地图、障碍物、小车模型、探索路径、frontier、任务状态、导航请求和系统事件。通过 WebSocket 接收后端状态快照，并结合 `mapDelta` 增量地图同步，可以实现较好的实时显示效果，满足仿真演示、状态监控和交互操作需求。

2）关键技术实现：

系统后端通过 FastAPI 提供登录认证、运行控制、地图配置、车辆配置、策略切换、状态查询和回放查询等接口。运行员可以启动、暂停、继续、停止和重置仿真；管理员可以管理用户；分析员可以查看历史回放，系统功能边界清晰。

系统支持两种运行形态。单机演示模式下，由 FastAPI 后端内嵌的 SimulationEngine 调度 Robot、Controller 和 Navigator，便于本地验收和稳定展示；分布式模式下，FastAPI 不直接执行仿真调度，而是由 Controller Worker、Navigator Worker 和 Robot Fleet Worker 独立运行。Controller 从 Redis 中读取开放 frontier、空闲车辆和任务状态，写入任务和导航请求；Navigator 从 Redis 中领取导航请求，规划路径后写回导航计划；Robot Fleet Worker 读取车辆、路径和地图状态，统一驱动 Redis 中配置好的全部小车，并写回车辆状态、地图 patch 和新发现的 frontier。

三类 worker 围绕 Redis 黑板协同工作，形成“扫描地图 -> 发现 frontier -> 分配任务 -> 规划路径 -> 执行路径 -> 继续扫描”的闭环。由于各组件之间只通过 Redis 共享数据，系统可以把接口服务、任务分配、路径规划和车辆执行拆分到不同进程或不同计算机上运行，具备明确的分布式实现基础，而不是停留在概念设计阶段。

路径规划与决策算法具备可实现性和可扩展性。系统支持最近可达 frontier 策略、信息增益贪心策略和 low_mdp 局部自主决策策略；导航模块支持 A*、时空 A* 和 CBS 冲突搜索算法。A* 和时空 A* 适合单车或带预约避让的路径规划，CBS 适合多车联合规划与冲突处理，能够满足多小车协作巡检场景下的核心算法需求。

数据一致性方面，系统将共享状态集中到 Redis 黑板中维护，地图版本、任务状态、导航请求、路径规划结果、组件心跳和事件日志都有明确的数据流向。后端在运行中限制地图、障碍和车辆数量等关键配置的随意修改，避免破坏正在执行的任务和路径；worker 通过心跳和事件记录上报运行状态，便于前端显示组件在线情况，也便于问题排查和恢复。

3）技术成熟度评估：

Python、FastAPI、Redis、Vue、Three.js、Docker 和 WebSocket 均为成熟且资料丰富的主流技术，学习成本和开发风险相对可控。A*、时空 A*、CBS、贪心策略和 MDP 等算法也具有明确的理论基础和实现路径。当前系统已经具备单机完整演示能力，并已完成基于 Redis 黑板的分布式 worker 拆分和启动脚本设计，可在“中心服务 + 远程 worker”的方式下运行。因此从技术实现、运行维护、组件解耦和后续扩展角度看，本系统具有较高的技术可行性。

## 1.4 经济可行性分析

1）硬件投入：

本系统主要用于多小车协作巡检的仿真演示和算法验证，对硬件要求不高。单机验收版本可在普通个人电脑上运行后端服务、Redis 和浏览器前端，无需额外采购专用设备。进行分布式展示时，也只需要两台普通计算机通过局域网连接，即可将前端、FastAPI、Redis 与 Controller、Navigator、Robot worker 分开部署，硬件投入较低。

系统采用开源技术栈进行开发，Python、FastAPI、Redis、Vue、Three.js、Docker 和 pytest 等工具均可免费使用，不需要支付商业软件授权费用。开发阶段可以利用现有代码框架和开源依赖完成接口服务、状态存储、三维渲染、算法实现和测试验证，从而降低开发成本。

维护阶段，系统采用模块化结构，前端、后端、黑板、Robot、Controller、Navigator 和算法策略之间职责清楚。若后续需要增加新的路径规划算法、任务分配策略或小车 worker，只需要在相应模块中扩展，不需要大范围修改整体系统，能够降低长期维护成本。

3）投资回报：

系统完成后可以用于课程验收、算法演示、软件构造设计说明和多车协作探索实验。与手工绘制流程或静态展示相比，系统能够通过三维界面实时展示车辆移动、地图覆盖率、任务队列和路径规划过程，明显提高展示效果和说服力。同时，该系统已经具备分布式部署基础，也为后续扩展真实小车控制、历史回放分析和多策略对比提供基础，具有较好的学习价值和扩展价值。因此从投入成本和产出效果看，系统具有良好的经济可行性。

## 1.5 社会因素可行性分析

### 1.5.1 法律因素

1）系统主要面向课程实验、仿真演示和算法研究，不涉及真实道路控制、公共安全调度或敏感生产数据，法律风险较低。

2）系统使用的主要技术组件和开发框架均为开源软件，项目开发过程中应遵守对应开源许可证要求。前端使用的车辆模型和素材也应保留其许可说明，避免未经授权的商业使用。

3）系统包含用户登录、角色权限和回放记录等数据，但这些数据主要用于本地演示和系统管理。若后续部署到真实网络环境，应遵守《网络安全法》《数据安全法》和《个人信息保护法》等相关要求，对账号密码、访问权限、运行日志和回放数据进行合理保护。

4）系统当前用于仿真场景，不直接采集真实个人隐私信息，也不对现实车辆或设备进行直接控制。因此在课程实验范围内，系统具有法律可行性。

### 1.5.2 用户使用可行性

1）系统提供浏览器端 Dashboard，用户不需要直接操作后端代码即可完成地图配置、障碍生成、车辆部署、策略选择、仿真启动、暂停、继续、重置和状态查看等操作，使用门槛较低。

2）系统按用户角色划分权限。超级管理员负责账号管理，系统运行员负责仿真配置和运行控制，分析员负责查看历史回放和分析结果。角色分工符合常见管理系统的使用习惯，有助于减少误操作。

3）前端界面能够实时展示地图、车辆、路径、frontier、任务、导航请求、覆盖率和事件日志，用户可以直观看到系统运行过程，不需要深入理解全部算法代码也能理解系统效果。

4）系统启动和演示流程较明确，后端文档中已经给出 Redis、Uvicorn 和 Dashboard 的启动方式。对于具备基础计算机操作能力的用户，只需按照说明启动服务并登录页面，即可完成主要操作。因此系统在用户使用层面具有可行性。

## 1.6 可供选择的方案

1）单机内嵌仿真方案：

该方案将前端页面、FastAPI 后端、Redis 黑板和 SimulationEngine 都部署在同一台计算机上，由后端内嵌调度 Robot、Controller 和 Navigator。优点是部署简单、启动流程清晰、演示稳定性较高，适合课程验收和本地调试；缺点是组件仍运行在同一台设备上，分布式特征不够明显，当车辆数量、地图规模或算法复杂度增加时，单机资源压力会增大。

2）分布式 Redis 黑板方案：

该方案将前端、FastAPI 和 Redis 黑板部署在中心计算机上，将 Robot、Navigator 等 worker 部署到另一台或多台计算机上。各组件通过 Redis 黑板交换状态，不直接相互调用。优点是低耦合、可扩展性强，能够更好地体现分布式系统设计思想，也便于增加小车数量或导航器数量；缺点是部署和调试复杂度更高，需要保证局域网连接、端口开放、Redis 地址配置和远程 worker 启动流程正确。

3）传统紧耦合单体方案：

该方案将小车移动、任务分配、路径规划和前端展示全部写在一个单体程序中，模块之间直接调用。优点是前期实现简单，适合快速做最小演示；缺点是后期维护困难，算法替换和功能扩展成本高，也不利于实现多机部署和组件独立运行。

综合比较，本系统推荐采用“单机可稳定演示、架构支持分布式扩展”的方案。当前验收阶段优先使用单机内嵌仿真保证系统完整运行；在设计说明和后续扩展中，保留基于 Redis 黑板的分布式 worker 部署能力。该方案兼顾实现稳定性、技术先进性和后续扩展性，是本系统较为合理的选择。
