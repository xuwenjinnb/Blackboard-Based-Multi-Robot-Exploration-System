# 接口设计讲解稿：从零理解本系统接口

这份讲稿对应项目的“3.5 接口设计”。如果你对接口完全不熟，可以先记住一句话：

> 接口就是不同模块之间约好的“说话方式”。前端通过 HTTP/WebSocket 跟 FastAPI 说话；后端、Controller、Navigator、Robot Worker 通过 Redis 黑板说话。

本项目接口代码主要看这几个文件：

| 看什么 | 代码位置 | 作用 |
| --- | --- | --- |
| 对外 HTTP 和 WebSocket 接口 | `backend/app/main.py` | 浏览器调用的 REST 接口、WebSocket 接口都在这里 |
| 前端怎么调用后端 | `frontend/Pathfinding-Visualizer-ThreeJS-master/src/services/api.js` | 前端 HTTP 请求封装 |
| 前端怎么连 WebSocket | `frontend/Pathfinding-Visualizer-ThreeJS-master/src/services/ws.js` | 前端实时接收状态快照 |
| Redis 黑板数据接口 | `backend/app/redis_blackboard.py` | 把系统状态真正写进 Redis |
| Worker 怎么接入 Redis | `backend/app/workers/common.py` 和各 worker 文件 | Controller/Navigator/Robot Worker 启动后连接 Redis |

## 1. 先弄懂四种“接口”

本系统里“接口”不是只有一种。

### 1.1 HTTP REST 接口

这是浏览器最常见的请求方式。

例如用户点击“启动仿真”，前端会发：

```text
POST /control/start
```

后端收到后，把系统状态改成 `RUNNING`。

你可以把它理解成：

```text
前端：我要启动
后端：好的，我把系统状态写成 RUNNING
```

### 1.2 WebSocket 接口

HTTP 更像“问一次答一次”。WebSocket 更像“电话一直开着”。

前端打开页面后连接：

```text
/ws?token=用户登录token
```

然后后端持续推送：

```text
STATE_SNAPSHOT
```

也就是当前地图、小车、任务、路径、事件等实时状态。前端不用一直手动刷新页面。

### 1.3 Redis 黑板接口

Redis 黑板不是给浏览器直接访问的。它是后端内部共享数据的地方。

可以把 Redis 想成办公室墙上的白板：

- Robot Worker 把“我在哪里、我扫描到了什么”写到白板上。
- Controller Worker 从白板上看“哪些车空闲、哪些 frontier 没人去”，然后写入任务。
- Navigator Worker 从白板上领取导航请求，规划路径后写回路径结果。
- FastAPI 后端从白板读出完整状态，再发给前端。

所以 Controller、Navigator、Robot 之间不是互相调用函数，而是都读写 Redis 黑板。

### 1.4 Worker 接入接口

Worker 本身一般不暴露 HTTP 服务。它们通过 `workers/common.py` 连接 Redis。

例如 Robot Worker 启动后：

```text
读取 Redis 黑板里的 system/vehicles/tasks
执行小车移动和扫描
写回 vehicles/map/frontiers/heartbeats
```

## 2. 一次完整调用是怎么流动的

以“点击启动”为例：

```text
用户点击启动
  -> 前端 api.js 调用 POST /control/start
  -> FastAPI main.py 收到请求
  -> main.py 调用 blackboard.set_system_status("RUNNING")
  -> Redis 黑板保存 system.status = RUNNING
  -> Worker 看到系统 RUNNING，开始工作
  -> FastAPI 通过 /ws 推送 STATE_SNAPSHOT
  -> 前端更新 3D 地图和面板
```

以“小车移动”为例：

```text
Robot Worker 读 Redis 中的小车任务
  -> RobotComponent 执行下一步
  -> 扫描地图
  -> upload_map_patch 写地图 patch
  -> update_vehicle_state 写小车新位置
  -> save_frontier 写新发现的 frontier
  -> WebSocket 推送新快照给前端
```

## 3. main.py 里的对外接口

`backend/app/main.py` 是后端接口总入口。你报告里写的 REST 和 WebSocket 基本都在这里。

| 接口 | 方法 | 谁调用 | 用途 |

代码位置：

```text

@app.get("/dashboard")

```

### 3.2 登录认证接口

| 接口 | 方法 | 前端函数 | 用途 |
| --- | --- | --- | --- |
| `/auth/login` | POST | `login()` | 登录，返回 token 和用户信息 |
| `/auth/me` | GET | `fetchCurrentUser()` | 根据 token 获取当前用户 |
| `/auth/logout` | POST | `logout()` | 退出登录，删除当前 session |

登录流程：

```text
用户输入账号密码
  -> POST /auth/login
  -> 后端检查 auth:users
  -> 成功后生成 token
  -> token 存入 Redis 的 auth:session:{token}
  -> 前端把 token 存到 localStorage
```

后续请求都会带：

```text
Authorization: Bearer token
```

通俗解释：

- token 就像临时门票。
- 登录成功后，前端以后每次进门都出示这张门票。
- 后端中间件检查门票是否有效、用户是什么角色。

代码位置：

```text
backend/app/main.py: /auth/login /auth/me /auth/logout
backend/app/auth.py: RedisAuthStore
frontend/.../src/services/api.js: login / fetchCurrentUser / logout
```

### 3.4 系统状态接口

| 接口 | 方法 | 前端函数 | 用途 |
| --- | --- | --- | --- |
| `/state` | GET | `fetchState()` | 获取当前完整系统状态 |
| `/runtime` | GET | `fetchRuntime()` | 获取当前运行配置 |

`/state` 返回什么：

```text
地图 map

frontier

导航请求 navigationRequests
```

```text

```

通俗解释：

- `/state` 是“整个系统现在长什么样”。
- `/runtime` 是“系统现在使用什么参数运行”。

代码位置：

```text
backend/app/main.py: @app.get("/state"), @app.get("/runtime")
frontend/.../src/services/api.js: fetchState(), fetchRuntime()
```

### 3.5 仿真控制接口

| 接口 | 方法 | 前端函数 | 用途 |
| --- | --- | --- | --- |
| `/control/start` | POST | `startSimulation()` | 启动仿真 |
| `/control/pause` | POST | `pauseSimulation()` | 暂停仿真 |
| `/control/resume` | POST | `resumeSimulation()` | 继续仿真 |
| `/control/stop` | POST | `stopSimulation()` | 停止仿真并结束回放 |
| `/control/reset` | POST | `resetSimulation()` | 重置运行状态 |

这些接口本质上主要改 `system.status`：

```text
RUNNING  表示正在运行
PAUSED   表示暂停
STOPPED  表示停止
RESET    表示重置
```

`/control/start` 还会做这些事：

1. 如果请求里带了 `policy`，更新任务分配策略。
2. 如果请求里带了 `navigatorAlgorithm`，更新导航算法。
3. 如果请求里带了 `navigatorCount`，更新导航器数量。
4. 确保有演示小车。
5. 把系统状态写成 `RUNNING`。
6. 开始记录回放。

通俗解释：

> 控制接口就像系统遥控器：启动、暂停、继续、停止、重置。

代码位置：

```text
backend/app/main.py: @app.post("/control/start") 到 @app.post("/control/reset")
frontend/.../src/services/api.js: startSimulation() 到 resetSimulation()
backend/app/replays.py: start()/record()/finish()
```

### 3.6 运行参数配置接口

| 接口 | 方法 | 前端函数 | 用途 |
| --- | --- | --- | --- |
| `/runtime/policy` | POST | `setPolicy()` | 设置控制器分配策略 |
| `/runtime/navigator` | POST | `setNavigatorAlgorithm()` | 设置导航器算法 |
| `/runtime/navigators` | POST | `setNavigatorCount()` | 设置导航器数量 |

#### `/runtime/policy`

请求示例：

```json
{
  "policy": "nearest"
}
```

作用：

- 设置 Controller 用什么策略分配 frontier。
- 常见值：`nearest`、`greedy`、`low_mdp`。

注意：

- 系统正在 `RUNNING` 时不能改。
- 要先暂停或停止。

#### `/runtime/navigator`

请求示例：

```json
{
  "navigatorAlgorithm": "baseline"
}
```

作用：

- 设置导航器路径规划模式。
- 常见值：`baseline`、`cbs`。

#### `/runtime/navigators`

请求示例：

```json
{
  "count": 3
}
```

作用：

- 设置逻辑导航器数量。
- 分布式演示时可以启动多个 Navigator Worker。

通俗解释：

> 这组接口不是控制小车立刻动，而是在设置“系统接下来按什么规则动”。

代码位置：

```text
backend/app/main.py: /runtime/policy /runtime/navigator /runtime/navigators
frontend/.../src/services/api.js: setPolicy / setNavigatorAlgorithm / setNavigatorCount
backend/app/workers/common.py: persist_runtime_config / read_runtime_config
```

### 3.7 小车和地图配置接口

#### `/runtime/vehicles`

```json
  "seed": 1
```

作用：

- 一次修改多个障碍格。

通俗解释：

代码位置：

### 3.8 回放接口

| 接口 | 方法 | 前端函数 | 用途 |
| --- | --- | --- | --- |
| `/replays` | GET | `fetchReplays()` | 获取回放列表 |
| `/replays/{replayId}` | GET | `fetchReplay()` | 获取某次回放详情 |

数据大概包括：

```text

```

代码位置：

```text

```

### 3.9 内部/调试 HTTP 接口

下面这些接口也在 `main.py` 里，但要注意：

> 分布式 Worker 正常运行时主要直接读写 Redis 黑板，不一定通过这些 HTTP 接口。它们更像是“把黑板方法包成 HTTP”的内部/调试接口。

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/robot/register` | POST | 注册单辆小车 |
| `/robot/state` | POST | 更新小车状态 |
| `/robot/map-patch` | POST | 上传小车扫描到的地图 patch |
| `/frontiers` | POST | 上传 frontier |
| `/navigation-requests` | POST | 创建导航请求 |
| `/navigation-requests/claim` | POST | 导航器领取导航请求 |
| `/navigation-plans` | POST | 写入路径规划结果 |
| `/vehicle-task` | GET | 查询某辆车当前任务 |
| `/heartbeat` | POST | 上报组件心跳 |

这些接口分别对应 Redis 黑板里的方法。

#### `/robot/register`

请求示例：

```json
{
  "vehicleId": "car-01",
  "pose": {
    "position": {"x": 2, "y": 2},
    "heading": 0
  }
}
```

作用：

- 把一辆小车写入黑板。
- 内部调用 `register_vehicle()`。

#### `/robot/state`

作用：

- 更新小车当前位置、朝向、状态、电量、当前任务等。
- 内部调用 `update_vehicle_state()`。

#### `/robot/map-patch`

作用：

- 小车扫描周围后，把看到的格子上传。
- 内部调用 `upload_map_patch()`。

#### `/frontiers`

作用：

- 保存新发现的 frontier。
- 内部调用 `save_frontier()`。

#### `/navigation-requests`

作用：

- 创建路径规划请求。
- 一般由 Controller 产生。
- 内部调用 `create_navigation_request()`。

#### `/navigation-requests/claim`

作用：

- Navigator 领取一个等待规划的请求。
- 内部调用 `claim_navigation_request()`。

#### `/navigation-plans`

作用：

- Navigator 写回规划结果。
- 成功时包含路径；失败时包含失败原因。
- 内部调用 `write_navigation_plan()`。

#### `/vehicle-task`

作用：

- Robot 查询自己当前任务。
- 内部调用 `get_vehicle_task()`。

#### `/heartbeat`

作用：

- Controller、Navigator、Robot 上报“我还活着、我现在是什么状态”。
- 前端能看到组件是否 READY、BUSY、OFFLINE。

## 5. api.js 和 ws.js 是干什么的

### 5.1 api.js

`api.js` 是前端访问 REST 接口的统一封装。

核心函数是：

```js
request(path, options)
```

它统一做三件事：

1. 拼接后端地址。
2. 自动带上 `Authorization: Bearer token`。
3. 如果请求失败，把错误信息抛给页面。

所以页面组件不需要每次都手写 `fetch()`。

例如：

```js
startSimulation()
```

内部就是：

```text
POST /control/start
```

### 5.2 ws.js

`ws.js` 是前端 WebSocket 封装。

它做这些事：

1. 用 token 连接 `/ws`。
2. 收到 `STATE_SNAPSHOT` 后调用 `onSnapshot()`。
3. 断线后自动重连。
4. 提供 `send()` 方法发送控制消息。

通俗解释：

> api.js 负责按钮请求，ws.js 负责实时画面。

## 6. RedisBlackboard 内部数据接口

RedisBlackboard 是内部最重要的数据接口。它不直接给浏览器调用，而是给 FastAPI 和 Worker 调用。

代码位置：

```text
backend/app/redis_blackboard.py
```

### 6.1 系统状态接口

| 方法 | 用途 |
| --- | --- |
| `set_system_status(status)` | 设置系统状态，如 RUNNING/PAUSED/STOPPED |
| `snapshot()` | 获取完整系统快照 |
| `snapshot_since(mapVersion, mapGeneration)` | 获取增量快照，减少传输量 |

通俗解释：

> 系统状态接口回答两个问题：系统现在跑没跑？系统当前所有数据是什么？

### 6.2 小车状态接口

| 方法 | 用途 |
| --- | --- |
| `register_vehicle(vehicle_id, pose)` | 注册小车 |
| `update_vehicle_state(data)` | 更新小车位置、朝向、状态、电量 |
| `get_vehicle_task(vehicle_id)` | 查某辆车当前任务 |

通俗解释：

> 小车状态接口就是管理“小车现在在哪里、在干什么、有没有任务”。

### 6.3 地图感知接口

| 方法 | 用途 |
| --- | --- |
| `upload_map_patch(patch)` | 上传小车扫描到的局部地图 |
| `configure_map(width, height, chunk_size)` | 配置地图大小 |
| `configure_obstacles(...)` | 配置真实障碍物 |

地图里常见状态：

```text
UNKNOWN   未知
FREE      已知可走
VISITED   小车走过
OBSTACLE  障碍物
RESERVED  保留/不可走
```

通俗解释：

> 小车不是一开始知道全地图，而是边走边扫描。扫描结果通过 map patch 一点点合并进黑板。

### 6.4 Frontier 接口

| 方法 | 用途 |
| --- | --- |
| `save_frontier(frontier)` | 保存新发现的 frontier |
| `open_frontiers()` | 读取所有还没被分配的 frontier |

frontier 是什么：

> 已知区域和未知区域的边界。小车去 frontier 附近扫描，通常能发现新地图。

### 6.5 任务接口

| 方法 | 用途 |
| --- | --- |
| `create_task_for_frontier(vehicle_id, frontier)` | 给小车创建探索任务 |
| `mark_task_done(task_id)` | 标记任务完成 |
| `mark_task_progress(task_id, index)` | 更新任务执行到路径第几步 |
| `get_vehicle_task(vehicle_id)` | 查询小车当前任务 |

通俗解释：

> 任务就是“car-01 去某个 frontier”。Controller 负责创建，Robot 负责执行。

### 6.6 导航请求接口

| 方法 | 用途 |
| --- | --- |
| `create_navigation_request(task, priority)` | 为任务创建路径规划请求 |
| `claim_navigation_request(navigator_id)` | 单个 Navigator 领取一个请求 |
| `claim_navigation_requests(navigator_id, limit)` | CBS 模式下一次领取多个请求 |

通俗解释：

> 导航请求就是 Controller 对 Navigator 说：请帮这辆车从当前位置规划到目标。

### 6.7 路径规划接口

| 方法 | 用途 |
| --- | --- |
| `write_navigation_plan(plan)` | 写入路径规划结果 |

规划成功时：

```text
status = SUCCESS
path = 路径点列表
```

规划失败时：

```text
status = NO_PATH
failReason = 失败原因
```

通俗解释：

> Navigator 算完路径后，把答案写回黑板。Robot 下一轮就能按 pathQueue 执行。

### 6.8 心跳接口

| 方法 | 用途 |
| --- | --- |
| `update_heartbeat(component_id, type, status, work_id, host, pid)` | 上报组件运行状态 |

心跳显示：

```text
Controller READY/BUSY
Navigator READY/BUSY
Robot READY/BUSY/OFFLINE
```

通俗解释：

> 心跳就是每个组件定期说一句“我还活着”。前端靠它判断组件有没有掉线。

### 6.9 事件接口

| 方法 | 用途 |
| --- | --- |
| `add_event(source, event_type, message)` | 记录系统事件 |

事件例子：

```text
系统启动
小车发现 frontier
导航器规划成功
小车遇到障碍
探索完成
```

通俗解释：

> 事件日志就是系统运行过程的“流水账”，方便演示和回放。

## 10. 演示时可以这样讲

可以直接照着这段说：

> 我们系统的接口分两层。第一层是对外接口，集中在 `main.py`，前端通过 `api.js` 调用 REST 接口完成登录、配置地图、小车配置、启动暂停和回放查询，通过 `ws.js` 连接 `/ws` 接收实时状态快照。第二层是内部数据接口，集中在 `redis_blackboard.py`，Controller、Navigator、Robot Worker 不直接互相调用，而是都读写 Redis 黑板。Robot 写小车状态、地图 patch 和 frontier；Controller 读空闲车辆和 frontier，写任务和导航请求；Navigator 领取导航请求，写回路径规划结果。FastAPI 再把 Redis 中的状态整理成快照，通过 WebSocket 推给前端，所以前端能实时显示地图、小车、任务、路径和事件。

## 11. 老师可能问的问题

### Q1：为什么不用 Controller 直接调用 Navigator？

因为直接调用会让组件强耦合。用了 Redis 黑板后，Controller 只负责写导航请求，Navigator 只负责领取请求。它们不需要知道对方进程在哪台电脑。

### Q2：前端为什么不直接访问 Redis？

浏览器直接访问 Redis 不安全，也不方便做权限控制。前端只能访问 FastAPI，FastAPI 检查 token 和角色后，再读写 Redis。

### Q3：REST 和 WebSocket 有什么区别？

REST 用来发控制命令，比如启动、暂停、配置地图；WebSocket 用来实时推送状态，比如小车位置变化、任务变化、路径变化。

### Q4：Worker 是不是 HTTP 服务？

不是。分布式 Worker 一般不对外提供 HTTP 接口，它们通过 Redis TCP 连接接入黑板。

### Q5：`/robot/register` 这些接口是不是 Worker 必须用？

不是必须。它们是把黑板方法包装成 HTTP 的内部/调试接口。真正的 Robot Worker 在代码里直接调用 `RedisBlackboard` 方法。

### Q6：Redis 黑板里面最核心的数据是什么？

最核心的是：

```text
map
vehicles
frontiers
tasks
navigation_requests
navigation_plans
heartbeats
events
```

这些数据组成了整个多小车探索系统的共享状态。

## 12. 一句话总结

> `main.py` 是对外服务窗口，`api.js/ws.js` 是前端打电话的工具，`redis_blackboard.py` 是内部共享白板，`workers/common.py` 是各个 Worker 接入白板的插头。整个系统靠这些接口把“前端展示、后端控制、Redis 状态中心、三类 Worker”连接成一个完整闭环。
