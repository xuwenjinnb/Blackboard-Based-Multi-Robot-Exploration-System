# 分布式低耦合改造设计

## 核心思路

两台电脑就够：

```text
电脑 A：FastAPI + 前端静态页面 + Redis 黑板
电脑 B：Controller Worker + Navigator Worker + Robot Fleet Worker
```

FastAPI 不直接调用 Controller、Navigator、Robot；三个 worker 之间也不互相直接调用。所有组件只通过 Redis 黑板交换状态。

```text
前端 -> FastAPI -> Redis
Controller 读 Redis：车辆、前沿、任务；写任务和导航请求
Navigator  读 Redis：导航请求；写导航路径
Robot      读 Redis：全部车辆和路径；写车辆状态、地图补丁、前沿
```

## 地图压缩存储

大地图下，性能瓶颈主要来自 Redis 黑板中地图 chunk 的读写体积。当前版本对 Redis 中的 `map:chunks` 做了存储层压缩：

```text
内存/接口格式：chunk.cells 仍然是普通 cells 列表，前端和算法无需改动
Redis 存储格式：每个 chunk 的 cells JSON 使用 zlib 压缩，再转为 base64 存储
编码标记：json-zlib-base64-v1
```

这样做的好处是对外数据结构保持兼容，同时减少 Redis 中地图 chunk 的字符串大小，降低大地图下 Redis 锁持有时间和网络传输压力。100x100 地图测试中，`map:chunks` 存储体积约为原始 chunk JSON 的 6.3%。

## 启动方式

电脑 A 启动 Redis 和 FastAPI，并关闭内嵌仿真循环：

```powershell
cd "C:\Users\Administrator\Desktop\Online continual learning\MOSE-master\3d\软件构造\实验\最后一次作业\backend"

docker compose -f docker-compose.redis.yml up -d
.\run_api_distributed.ps1 -HostAddress 0.0.0.0 -Port 8000 -BroadcastIntervalSeconds 0.15
```

电脑 B 连接电脑 A 的 Redis，分别启动三个后端组件：

```powershell
cd "电脑B上的项目路径\backend"

$redis = "redis://电脑A的IP:6379/0"

.\run_controller_worker.ps1 -RedisUrl $redis
.\run_navigator_worker.ps1 -RedisUrl $redis -NavigatorIds "navigator-01,navigator-02" -BatchesPerTick 3
.\run_robot_worker.ps1 -RedisUrl $redis -StepsPerTick 1
```

注意：Robot 是一个 fleet worker，内部管理 Redis 中已经配置好的全部小车。不要再给每辆小车分别启动一个 `robot_worker`。

## 前端配置

浏览器打开：

```text
http://电脑A的IP:8000/dashboard
```

在前端配置地图、障碍物和车辆数量。车辆列表由 FastAPI 写入 Redis，Robot fleet worker 会自动读取并驱动所有车辆。

## 为什么这是分布式

```text
前端、FastAPI、Redis 在电脑 A；
Controller、Navigator、Robot 在电脑 B；
组件可以在不同进程、不同终端、不同电脑上运行；
组件之间只读写 Redis 黑板，不直接依赖彼此进程地址。
```

答辩时可以这样说：

```text
本系统采用 Redis 黑板作为共享状态中心。FastAPI 只负责前端入口、运行配置和状态接口，不执行仿真调度。Controller、Navigator 和 Robot 被拆成独立 worker，分别负责任务分配、路径规划和车辆执行。三个 worker 之间不直接通信，只通过 Redis 交换任务、路径、车辆状态和地图更新，因此系统具备分布式部署和低耦合扩展能力。
```
