# 多小车协作巡检仿真系统流程图设计稿

## 1. 图的定位

建议图名：

```text
分布式多小车协作巡检仿真系统流程图（多泳道图）
```

这张图要表达的不是单个算法细节，而是“前端控制、中心调度、Redis 黑板、Robot 小车、Controller 控制器、Navigator 导航器”如何通过黑板解耦协作，完成多车探索、任务分配、路径规划、执行反馈和实时可视化。

当前项目依据：

- `frontend/index.html`：前端有地图配置、障碍部署、车辆部署、任务分配策略、路径规划算法、启动/暂停/重置、车辆/任务/导航器/事件面板。
- `frontend/app.js`：前端通过 `/state`、`/ws`、`/control/*`、`/runtime/*` 与后端交互，并渲染 3D 地图、车辆、frontier、路径和事件。
- `backend/app/main.py`：FastAPI 提供控制接口、运行配置接口、WebSocket 快照推送，并创建 `RedisBlackboard` 与 `SimulationEngine`。
- `backend/app/simulation.py`：仿真 tick 中按顺序执行 Robot、Controller、Navigator；`low_mdp` 模式下由 Robot 直接决策。
- `backend/app/blackboard.py` 与 `redis_blackboard.py`：黑板保存地图、车辆、frontier、任务、导航请求、路径计划、心跳和事件。
- `backend/app/controller.py` 与 `policies.py`：控制器读取 frontier 和空闲车辆，按 nearest 或 greedy 策略创建任务和导航请求。
- `backend/app/robot/component.py`：Robot 执行路径、扫描、上传地图 patch、发现 frontier、阻塞时触发重规划。
- `backend/app/navigator/component.py`：Navigator 领取导航请求，用 A*、时空 A* 或 CBS 生成路径计划。
- `算法/后端算法整理.md`：已经整理了控制器、导航器和小车算法，可作为图中文字和答辩说明依据。

## 2. 推荐泳道设计

参考图是“界面 / 控制器 / 小车 / 导航器”的四泳道。你的项目建议扩展为六泳道，因为现在有明确的 FastAPI 调度层和 Redis 黑板层：

| 泳道 | 建议标题 | 图中承担的角色 |
| --- | --- | --- |
| 1 | 前端界面 | 用户配置地图、障碍、车辆、算法，启动/暂停/重置仿真，并实时显示 3D 地图、指标和事件 |
| 2 | FastAPI / 调度器 | 接收控制命令，校验运行状态，更新运行配置，启动 SimulationEngine tick，向前端推送快照 |
| 3 | Redis 黑板 | 系统共享状态中心，保存 map、vehicles、frontiers、tasks、navigationRequests、navigationPlans、heartbeats、events |
| 4 | 小车 Robot | 注册和心跳，执行路径，扫描局部地图，上传 map patch，发现 frontier，阻塞时上报重规划 |
| 5 | 控制器 Controller | 读取 OPEN frontier 和 IDLE 车辆，执行任务分配策略，创建任务和导航请求 |
| 6 | 导航器 Navigator | 领取导航请求，执行 A* / 时空 A* / CBS，写回 SUCCESS 或 NO_PATH 路径计划 |

版面建议采用横向 16:9 或更宽画布。Redis 黑板放在中间泳道，突出“组件不直接互调，只读写黑板”的低耦合特点。

## 3. 主流程内容设计

### 3.1 启动与配置链路

这部分放在图的左上角，从“前端界面”泳道开始：

1. 用户打开 Dashboard。
2. 配置地图大小、分块、障碍数量、车辆数量。
3. 选择任务分配策略：`nearest`、`greedy`、`low_mdp`。
4. 选择路径规划算法：`baseline` 或 `cbs`。
5. 点击启动。
6. FastAPI 接收 `/control/start`。
7. 如果系统正在运行，则拒绝修改配置；否则写入运行配置。
8. SimulationEngine 进入 tick 循环。
9. Redis 黑板记录 `systemStatus = RUNNING` 和系统事件。

图中可以把“系统正在运行？”画成粉色判定菱形；“写入运行配置”和“进入 tick 循环”画成蓝色流程块。

### 3.2 普通探索链路

普通模式指 `policy != low_mdp`。这是图的主干：

1. Robot 读取自己的任务。
2. 如果没有任务，则进入扫描状态。
3. Robot 按扫描半径观察周围格子。
4. Robot 上传 map patch。
5. Redis 黑板合并地图 patch，更新 map version 和 mapDelta。
6. Robot 检测 frontier。
7. Redis 黑板保存 OPEN frontier。
8. Controller 读取 OPEN frontier 和 IDLE 车辆。
9. Controller 按策略选择“哪辆车去哪个 frontier”。
10. Controller 创建 task，并把 frontier 标记为 ASSIGNED。
11. Controller 创建 navigation request。
12. Navigator 领取 PENDING 请求。
13. Navigator 根据算法生成路径计划。
14. Redis 黑板写入 navigation plan，并把 path 放入 task.pathQueue。
15. Robot 读取 pathQueue 并执行下一步。
16. Robot 移动后继续扫描，形成循环。

这部分建议作为主图中间最粗的蓝色流程线，从 Robot 到 Redis，再到 Controller，再到 Redis，再到 Navigator，再回到 Redis 和 Robot。

### 3.3 路径执行与反馈链路

这部分放在 Robot 泳道中下部，并与 Redis 黑板形成回路：

1. Robot 判断下一步是否可走。
2. 如果下一步在真实障碍中：
   - 扫描并上传地图。
   - 上报 `VEHICLE_BLOCKED`。
   - task 重新置为 `PENDING`。
   - 清空旧路径并增加 `replanCount`。
   - Controller 下一轮补发导航请求。
3. 如果下一步可走：
   - 更新车辆位置、朝向、电量、任务进度。
   - 扫描并上传地图。
4. 如果路径走完：
   - task 标记为 `DONE`。
   - frontier 标记为 `VISITED`。
   - 车辆回到 `IDLE`。

图中“下一步被阻塞？”和“路径是否走完？”建议用粉色判定菱形；重规划回路用红色或粉色虚线箭头标出。

### 3.4 导航规划链路

Navigator 泳道建议把 baseline 与 CBS 画成一个分支：

1. 判断导航算法。
2. `baseline`：
   - 每个 navigator 领取一个 PENDING 请求。
   - 如果需要避让车辆，则使用时空 A*。
   - 否则使用普通 A*。
3. `cbs`：
   - 第一个 navigator 批量领取多个请求。
   - CBS 检测顶点冲突和边冲突。
   - 对冲突车辆加约束并重规划。
4. 判断是否找到路径。
5. 成功则写入 `SUCCESS plan`。
6. 失败则写入 `NO_PATH` 和失败原因。

图中不要展开所有 A* 公式，只保留算法选择、规划、成功/失败三层。公式和参数可以放在报告正文，不塞进流程图。

### 3.5 `low_mdp` 特殊路径

`low_mdp` 是一条特殊支线，建议从 FastAPI / 调度器泳道的“policy = low_mdp？”判定菱形引出，直接连到 Robot：

1. Robot 读取地图和车辆状态。
2. 构造局部可通行状态空间。
3. 计算未知格子奖励和多车斥力。
4. 价值迭代选择下一步。
5. Robot 移动或等待。
6. 扫描并上传地图。
7. 如果没有 UNKNOWN 格子，则暂停仿真。

这条线要标注“跳过 Controller 与 Navigator”，因为 `low_mdp` 下控制器使用 Noop 策略，不创建分配结果，导航器也不参与全局路径规划。

### 3.6 实时可视化链路

这部分从 Redis 黑板经 FastAPI 回到前端：

1. Redis 黑板保存最新 mapDelta、车辆、frontier、任务、计划、心跳、事件。
2. FastAPI 的 `/state` 和 `/ws` 生成状态快照。
3. 前端合并全量地图或增量 mapDelta。
4. 3D 地图刷新格子、车辆模型、frontier、路径。
5. 指标面板更新覆盖率、行走步数、车辆数、frontier、任务和导航请求。

图中这条可以用一条顶部或底部的回流箭头表示：“状态快照 / WebSocket 推送”。

## 4. 可选分布式部署支线

如果你想更贴近“分布式低耦合改造设计.md”，可以在主图右上角或下方加一个虚线支线：

```text
前端选择 host-b 创建小车/导航器
  -> FastAPI 写入创建命令
  -> Redis 命令队列
  -> Host Agent 读取命令
  -> 电脑 B 启动 robot_worker / navigator_worker
  -> Worker 写入 heartbeat
```

注意：这条属于设计目标或扩展设计。如果当前代码中还没有完整 Host Agent / worker 实现，图里应使用虚线边框，并标注“扩展设计”或“可选分布式部署”。

## 7. 最终绘图版式建议

### 7.1 画布与泳道

- 画布：横向 16:9，建议 1920x1080 或 2560x1440。
- 顶部标题栏：洋红色，文字白色，居中写“分布式多小车协作巡检仿真系统流程图”。
- 背景：浅蓝色或浅青色，与参考图保持一致。
- 泳道：六列等宽或 Redis 黑板列略窄；每列顶部有深粉标题条。
- 分隔线：浅蓝或灰蓝细线。

### 7.2 颜色与形状

| 类型 | 形状 | 建议颜色 | 用途 |
| --- | --- | --- | --- |
| 开始 / 结束 | 圆角胶囊 | 洋红 `#d80072` | 打开 Dashboard、探索完成 |
| 普通流程 | 矩形或小圆角矩形 | 蓝色 `#139bd7` | 接收命令、扫描、创建任务、规划路径 |
| 判断条件 | 菱形 | 洋红 `#e00070` | 系统是否运行、是否 low_mdp、是否有任务、是否找到路径 |
| Redis 数据 | 圆柱或数据库图标 | 青绿色 `#23b5aa` | 黑板中的共享数据 |
| 可选扩展 | 虚线框 | 浅灰 | Host Agent、远程 worker 创建 |
| 异常 / 重规划 | 粉色虚线箭头 | 洋红或红色 | 阻塞上报、重规划回路 |

### 7.3 箭头语义

- 实线箭头：主流程和数据写入。
- 虚线箭头：可选扩展、状态推送、重规划反馈。
- 箭头尽量从左到右；循环反馈可以走底部或顶部，避免穿过太多节点。
- Redis 黑板相关箭头可以写短标签：`写入 patch`、`读取 frontier`、`写入 request`、`写入 plan`、`推送 snapshot`。

### 7.4 字体与标签密度

- 每个节点尽量 6 到 14 个汉字，最多两行。
- 不要把代码函数名大量塞进节点，可以在节点下方小字标注少量关键名，例如 `pathQueue`、`mapDelta`、`navigationRequests`。
- 公式、默认参数、训练细节不放进主流程图。
- A*、CBS、low_mdp 这些算法名需要保留，因为它们是系统可选能力。

## 8. 建议最终主图结构

最终画出来时，可以按以下顺序组织画面：

1. 顶部：标题栏。
2. 左上：前端启动与运行配置。
3. 中上：FastAPI 接收命令，写入配置，启动 tick。
4. 中央：Redis 黑板作为共享状态中心。
5. 中下：普通模式主循环：Robot 扫描 -> Controller 分配 -> Navigator 规划 -> Robot 执行。
6. 右下：阻塞上报与重规划回路。
7. 底部：状态快照通过 `/ws` 回到前端。
8. 右上或底部角落：Host Agent / 远程 worker 创建，使用虚线表示扩展设计。

## 9. 不建议放进图里的内容

- 每一个 FastAPI endpoint 的完整名称。
- A*、CBS、MDP 的详细公式和参数。
- 前端 Three.js 渲染细节、车辆模型资源路径、贴图资源。
- Redis 每个 key 的实际命名细节。
- 算法实现中的低层参数细节。

这些内容更适合放在报告正文或附录，流程图只保留系统协作主线。

## 10. 一句话答辩说明

可以配合图这样讲：

```text
本系统采用 Redis 黑板作为共享状态中心，前端通过 FastAPI 控制仿真并实时接收状态快照；Robot 负责执行、感知和 frontier 发现，Controller 负责多车任务分配，Navigator 负责路径规划。各组件不直接调用彼此，而是通过黑板读写 map、frontier、task、request、plan、heartbeat 和 event，从而实现多车协作、低耦合通信和可扩展的分布式部署。
```
