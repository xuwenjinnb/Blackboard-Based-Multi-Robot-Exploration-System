# 多小车协作巡检仿真系统数据流图设计整理

本文根据当前项目代码整理数据流图需要的“数据存储、实体、一层子系统、二层加工”。阅读依据主要包括：

- `backend/app/main.py`：FastAPI 接口、认证权限、运行控制、WebSocket 快照、回放接口。
- `backend/app/redis_blackboard.py`：Redis 黑板的 key、Hash/Set/Stream/List 持久化方式。
- `backend/app/blackboard.py`：地图、小车、frontier、任务、导航请求、导航计划等实体字段。
- `backend/app/auth.py`：用户、会话、角色权限。
- `backend/app/replays.py`：回放索引与回放帧。
- `backend/app/simulation.py`、`controller.py`、`robot/component.py`、`navigator/component.py`：Robot、Controller、Navigator 的运行闭环。
- `frontend/Pathfinding-Visualizer-ThreeJS-master/src/components/*.vue`：登录、用户管理、运行控制台、回放分析界面。

## 1. Redis 里的数据是不是“表”

严格说，Redis 不是关系型数据库，没有 SQL 表、主外键和固定 schema。但在软件工程的数据流图里，可以把 Redis 中的 Hash、List、Set、Stream、String 统一画成“数据存储”，也可以在报告里称为“Redis 逻辑表”。

建议写法：

```text
本系统使用 Redis 黑板保存共享状态。Redis 中的数据不是传统关系表，但可按数据流图中的数据存储处理，其中 Hash 可近似理解为以 field 为主键、JSON 为记录体的逻辑表。
```

画 DFD 时，不建议把所有 Redis 技术 key 都展开成十几个数据库图标。一层图里可以合并成 5 到 7 个数据存储；二层图或说明文字中再列出具体 Redis key。

## 2. Redis 逻辑表 / 数据存储

默认前缀来自 `RedisConfig.prefix`，代码里默认为 `inspection`。下面用 `inspection:*` 表示实际 Redis key。

| 编号 | Redis key / 类型 | 可画成的逻辑表 | 主要实体 | 说明 |
| --- | --- | --- | --- | --- |
| D1 | `inspection:auth:users` / Hash | 用户表 | User | field 为 username，value 为用户 JSON，保存账号、角色、启用状态和密码哈希。 |
| D2 | `inspection:auth:session:{token}` / String with TTL | 登录会话表 | LoginSession | 12 小时过期，用于 REST 和 WebSocket 权限校验。DFD 中可并入用户表。 |
| D3 | `inspection:runtime` / Hash | 运行配置表 | RuntimeConfig | 保存任务分配策略、导航算法、导航器数量和更新时间。 |
| D4 | `inspection:system` / Hash | 系统状态表 | SystemStatus | 保存 `STOPPED`、`RUNNING`、`PAUSED` 等运行状态。 |
| D5 | `inspection:map:meta` / Hash | 地图元数据表 | Map | 保存 mapId、宽高、chunkSize、version、generation、updatedAt。 |
| D6 | `inspection:map:chunks` / Hash | 地图分块表 | MapChunk、MapCell | field 为 chunkId，value 为分块及格子列表，是当前主要地图存储。 |
| D7 | `inspection:true_obstacles` / Set | 真实障碍表 | Obstacle | 保存真实障碍坐标，成员格式为 `x:y`。 |
| D8 | `inspection:vehicles` / Hash | 小车状态表 | Vehicle | 保存小车位置、朝向、电量、状态、当前任务、当前路径进度。 |
| D9 | `inspection:frontiers` / Hash | 前沿点表 | Frontier | 保存待探索前沿点、得分、未知收益、发现者和状态。 |
| D10 | `inspection:tasks` / Hash | 巡检任务表 | InspectionTask | Controller 创建，Robot 执行，Navigator 规划后写入 pathQueue。 |
| D11 | `inspection:navigation_requests` / Hash | 导航请求表 | NavigationRequest | Controller 或重规划流程写入，Navigator 领取。 |
| D12 | `inspection:navigation_plans` / Hash | 路径规划表 | NavigationPlan | Navigator 写回 `SUCCESS` 或 `NO_PATH` 结果。 |
| D13 | `inspection:heartbeats` / Hash | 组件心跳表 | ComponentHeartbeat | 保存 Robot、Controller、Navigator 的在线状态、主机、pid、当前工作。 |
| D14 | `inspection:events` / Stream | 系统事件流 | SystemEvent | 保存地图更新、任务分配、路径成功/失败、阻塞、控制命令等事件。 |
| D15 | `inspection:counters` / Hash | 编号计数表 | IdCounter | 生成 event、task、frontier、request、plan、patch 等递增编号。 |
| D16 | `inspection:replays:index` / Hash | 回放索引表 | ReplayMeta | 保存每次运行回放的元信息，如操作员、策略、帧数、覆盖率。 |
| D17 | `inspection:replays:frames:{replayId}` / List | 回放帧表 | ReplayFrame | 保存压缩后的历史快照帧。 |

可以忽略或只在说明中提到的技术 key：

- `inspection:lock`：Redis 黑板全局写锁。
- `inspection:locks:controller`：分布式 Controller 抢占锁。
- `inspection:initialized`、`inspection:true_obstacles:initialized`：初始化标记。
- `inspection:map:cells`：旧版兼容地图格子存储，当前主要使用 `map:chunks`。

## 3. 核心实体

| 实体 | 关键字段 | 来源 | 去向 / 被谁使用 |
| --- | --- | --- | --- |
| User 用户 | username、displayName、role、enabled、salt、passwordHash、createdAt、updatedAt | 管理员创建或系统默认初始化 | 登录认证、权限控制、用户管理 |
| LoginSession 登录会话 | token、username、createdAt、TTL | 登录成功后生成 | REST 接口鉴权、WebSocket 鉴权 |
| RuntimeConfig 运行配置 | policy、navigatorAlgorithm、navigatorCount、updatedAt | 运行员在前端配置 | FastAPI、SimulationEngine、Worker |
| SystemStatus 系统状态 | status | 控制命令 start/pause/resume/stop/reset | SimulationEngine、worker、前端状态展示 |
| Map 地图 | mapId、width、height、chunkSize、version、generation、updatedAt | 配置地图或地图 patch 更新 | Robot 扫描、Controller 评估、Navigator 规划、前端渲染 |
| MapCell 地图格子 | x、y、state、confidence、updatedBy、updatedAt | Robot 扫描、障碍配置 | 地图分块、frontier 检测、路径规划、回放 |
| Obstacle 障碍 | x、y、blocked | 运行员手动/随机配置 | Robot 阻塞判断、地图显示、导航规划 |
| Vehicle 小车 | vehicleId、pose、speed、battery、status、currentTaskId、currentPlanId、currentStepIndex、updatedAt | SimulationEngine 初始化、Robot 更新 | Controller 分配任务、Navigator 避让、前端显示 |
| Frontier 前沿点 | frontierId、position、unknownGain、discoveredBy、status、timestamp、updatedAt | Robot 扫描发现 | Controller 任务分配 |
| InspectionTask 巡检任务 | taskId、vehicleId、target、frontierId、status、planId、pathQueue、currentStepIndex、replanCount、createdAt、updatedAt | Controller 创建 | Navigator 规划、Robot 执行 |
| NavigationRequest 导航请求 | requestId、vehicleId、taskId、start、goal、mapVersion、priority、avoidVehicles、status、assignedNavigatorId、createdAt、updatedAt | Controller 创建或阻塞重规划 | Navigator 领取并规划 |
| NavigationPlan 导航计划 | planId、requestId、taskId、vehicleId、start、goal、path、cost、distance、estimatedTime、mapVersion、status、planner、createdBy、createdAt、failReason | Navigator 写回 | Task.pathQueue、Robot 执行、前端显示 |
| ComponentHeartbeat 组件心跳 | componentId、componentType、status、currentWorkId、updatedAt、host、pid | Robot/Controller/Navigator worker | 运行状态面板、分布式部署监控 |
| SystemEvent 系统事件 | eventId、source、type、message、createdAt | 各组件写入 | 前端事件面板、回放记录 |
| ReplayMeta 回放索引 | replayId、operator、status、startedAt、endedAt、frameCount、policy、navigatorAlgorithm、vehicleCount、mapWidth、mapHeight、lastCoverage、movementSteps | 控制开始/停止与广播循环 | 分析员回放列表 |
| ReplayFrame 回放帧 | timestamp、snapshot | 运行时定时记录 | 回放播放、统计分析 |

角色也可以作为实体或枚举处理：

- `admin`：超级管理员，进入用户管理。
- `operator`：系统运行员，配置和控制仿真。
- `analyst`：分析员，查看历史回放。

| P4 | 运行控制与调度子系统 | start/pause/resume/stop/reset 命令 | D3 运行配置、D4 系统状态、D8 小车 | D4 系统状态、D13 心跳、D14 事件、D16/D17 回放 | 运行状态、调度 tick |

| P8 | 状态快照与实时可视化子系统 | 前端 `/state` 或 `/ws` 请求 | D3-D14 当前黑板状态 | 无，或写少量连接状态 | 状态快照、mapDelta、WebSocket 推送 |
| P9 | 回放记录与分析子系统 | 运行过程快照、分析员回放请求 | D16 回放索引、D17 回放帧 | D16/D17 | 回放列表、回放帧、统计数据 |

一层图中的外部实体建议画：

- E1 超级管理员：登录、管理用户。
- E2 系统运行员：登录、配置并控制仿真、查看实时状态。
- E3 分析员：登录、查看历史回放。
- E4 分布式 Worker 进程：如果要突出低耦合部署，可把 Controller Worker、Navigator Worker、Robot Worker 画成系统内部加工；如果要突出跨机器部署，也可在图边界外画成外部执行节点，但仍然只通过 Redis 黑板交换数据。

一层图中的数据存储建议合并画：

- D1 认证与用户数据：对应 `auth:users`、`auth:session:*`。
- D2 运行配置与系统状态：对应 `runtime`、`system`。
- D3 地图与障碍数据：对应 `map:meta`、`map:chunks`、`true_obstacles`。
- D4 车辆与组件状态：对应 `vehicles`、`heartbeats`。
- D5 任务与导航数据：对应 `frontiers`、`tasks`、`navigation_requests`、`navigation_plans`。
- D6 事件数据：对应 `events`。
- D7 回放数据：对应 `replays:index`、`replays:frames:*`。

## 6. 推荐画图层次

### 6.1 0 层图 / 上下文图

如果报告要求先画顶层上下文图，可以只画一个总加工：

```text
E1 超级管理员
E2 系统运行员
E3 分析员
        -> P0 多小车协作巡检仿真系统
        -> D Redis 黑板 / 回放数据
```

数据流：

- 超级管理员 -> 系统：登录信息、用户管理请求。
- 系统 -> 超级管理员：登录结果、用户列表、管理结果。
- 系统运行员 -> 系统：仿真配置、控制命令。
- 系统 -> 系统运行员：实时状态、地图、小车、任务、路径、事件。
- 分析员 -> 系统：回放查询请求。
- 系统 -> 分析员：回放列表、回放帧、统计指标。

### 6.2 一层图

一层图画 P1-P9 九个子系统即可。不要在一层图里展开 A*、CBS、低层 MDP 的内部算法，否则图会过细。

推荐主线：

```text
系统运行员 -> P3 仿真配置 -> D2/D3/D4
系统运行员 -> P4 运行控制与调度 -> P5 Robot -> D3/D4/D5
D5 frontiers/tasks -> P6 Controller -> D5 navigation_requests
D5 navigation_requests -> P7 Navigator -> D5 navigation_plans/tasks
D2-D6 -> P8 状态快照与可视化 -> 系统运行员
P8 / P4 -> P9 回放记录与分析 -> 分析员
超级管理员 -> P2 用户管理 -> D1
所有用户 -> P1 登录认证与权限控制 -> D1/D2
```

### 6.3 二层图

二层图建议画 4 张：

1. `P1 登录认证与权限控制二层图`
2. `P3 仿真配置二层图`
3. `P4-P7 巡检运行核心闭环二层图`
4. `P9 回放记录与分析二层图`

其中第 3 张是最重要的，可以把 Robot、Controller、Navigator 分成三列，中间放 Redis 黑板数据存储。

## 8. 最终结论

1. Redis 里保存的数据可以在 DFD 中当作“表”或“数据存储”画，但报告中最好称为“Redis 逻辑表 / 黑板数据存储”。
2. 一层图建议画 9 个子系统：登录认证、用户管理、仿真配置、运行控制、Robot、Controller、Navigator、状态可视化、回放分析。
3. 二层图重点展开登录认证、仿真配置、巡检运行核心闭环、回放分析。
4. 核心运行闭环是：Robot 写地图和 frontier，Controller 读 frontier 和车辆后写任务与导航请求，Navigator 读请求后写路径计划，Robot 再读取 pathQueue 执行，前端通过快照实时展示。
5. 图中最能体现项目特色的是 Redis 黑板：Robot、Controller、Navigator 互不直接调用，只通过 Redis 数据存储读写状态。
