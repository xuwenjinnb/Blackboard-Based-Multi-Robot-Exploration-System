# 多小车协作巡检仿真系统多泳道图设计分析

## 1. 修改结论

你这张多泳道图应该借鉴学长学姐那种画法，但不能完全照搬“四个技术泳道”。你当前项目已经有登录、三类用户、权限控制、仿真运行、历史回放，所以更适合画成：

```text
界面 / 后台服务与Redis黑板 / 控制器 / 小车 / 导航器
```

其中“界面”泳道里必须把三个角色画出来：

```text
超级管理员
系统运行员
分析员
```

也就是说，最终图不是单纯的“控制器、小车、导航器怎么跑”，而是先从“用户登录和角色分流”开始，再进入对应业务流程。

## 2. 项目依据

我根据你当前项目代码梳理，多泳道图应体现这些真实功能。

| 功能 | 代码位置 | 图中怎么体现 |
| --- | --- | --- |
| 登录系统 | `frontend/Pathfinding-Visualizer-ThreeJS-master/src/components/LoginView.vue` | 界面泳道画“显示登录界面”“输入账号密码”“合法用户？” |
| 三角色分流 | `frontend/Pathfinding-Visualizer-ThreeJS-master/src/App.vue` | 登录成功后判断 admin / operator / analyst |
| 超级管理员管理用户 | `frontend/.../AdminUsers.vue`、`backend/app/main.py` 的 `/admin/users` | 超级管理员分支画“用户管理：增删改查” |
| 系统运行员运行仿真 | `frontend/.../PathfindingVisualizer.vue`、`backend/app/main.py` 的 `/runtime/*`、`/control/*` | 系统运行员分支画“配置地图、车辆、算法、启动仿真” |
| 分析员查看回放 | `frontend/.../ReplayViewer.vue`、`backend/app/replays.py`、`/replays` | 分析员分支画“选择回放、播放回放、查看数据” |
| 权限控制 | `backend/app/main.py` 中 `authorize_request` | 后台服务泳道画“校验 token 和角色权限” |
| Redis 黑板 | `backend/app/redis_blackboard.py` | 后台服务与 Redis 黑板泳道画共享状态 |
| 调度顺序 | `backend/app/simulation.py` | 运行员启动后画 Robot -> Controller -> Navigator 循环 |
| 控制器任务分配 | `backend/app/controller.py`、`backend/app/policies.py` | 控制器泳道画读取 frontier、空闲车、创建任务 |
| 小车执行探索 | `backend/app/robot/component.py` | 小车泳道画扫描、上传地图、发现 frontier、执行路径 |
| 导航器路径规划 | `backend/app/navigator/component.py`、`backend/app/navigator/planners/*` | 导航器泳道画领取请求、A*/CBS、写回路径 |

## 3. 推荐最终泳道

建议画 5 个泳道，既像参考图，又能说明你项目的 Redis 黑板。

| 泳道 | 建议标题 | 作用 |
| --- | --- | --- |
| 1 | 界面 | 登录、角色分流、用户管理、仿真配置、回放查看、实时显示 |
| 2 | 后台服务 / Redis 黑板 | 校验权限、接收前端命令、保存共享状态、记录回放、推送快照 |
| 3 | 控制器 | 读取 frontier 和空闲车辆，执行任务分配策略，创建任务和导航请求 |
| 4 | 小车 | 扫描地图、上传地图 patch、发现 frontier、执行路径、反馈阻塞和完成状态 |
| 5 | 导航器 | 领取导航请求，执行 A* / 时空 A* / CBS，写回路径规划结果 |

如果老师更喜欢参考图那种四泳道，也可以把“后台服务 / Redis 黑板”缩成几个数据块放在“控制器、小车、导航器”之间，但我更建议单独保留，因为这是你项目的核心亮点。

## 4. 界面泳道必须画三角色

参考学长学姐的图，左侧“界面”泳道应该这样设计：

```text
开始
  ↓
显示登录界面
  ↓
输入账号密码
  ↓
合法用户？
  ├─ 否 -> 提示账号或密码错误
  └─ 是 -> 判断登录身份
          ├─ 超级管理员？
          ├─ 系统运行员？
          └─ 分析员？
```

三个角色分别走三条分支。

### 4.1 超级管理员分支

超级管理员默认账号：

```text
huadian / 123456
```

图中流程：

```text
超级管理员身份登录？
  ↓ 是
进入用户管理界面
  ↓
查看系统用户列表
  ↓
新增 / 编辑 / 删除用户
  ↓
选择用户角色：系统运行员 / 分析员
  ↓
后台写入用户数据
  ↓
返回用户管理结果
```

可以写进图中的节点文字：

```text
进入用户管理界面
查看系统账户
新增用户
编辑用户
删除用户
分配角色
```

这条分支不需要进入小车、控制器、导航器流程。

### 4.2 系统运行员分支

系统运行员默认账号：

```text
operator / 123456
```

这是主流程，需要接到控制器、小车、导航器。

图中流程：

```text
系统运行员身份登录？
  ↓ 是
进入仿真运行界面
  ↓
配置地图大小、分块、障碍
  ↓

  ↓

  ↓

  ↓
点击启动探索
  ↓
后台服务接收启动命令
  ↓
Redis 黑板写入 RUNNING 和运行配置
  ↓
小车 / 控制器 / 导航器进入协作循环
  ↓
界面实时显示地图、车辆、任务、路径、事件
```

可以写进图中的节点文字：

```text
配置地图

配置小车数量
选择任务分配策略
选择路径规划算法
点击开始探索
实时显示探索过程
暂停 / 继续 / 重置
```

### 4.3 分析员分支

分析员默认账号：

```text
analyst / 123456
```

图中流程：

```text
分析员身份登录？
  ↓ 是

  ↓
查看历史回放列表
  ↓

  ↓
播放 / 暂停 / 调整倍速
  ↓
查看车辆数据和本帧黑板数据
```

```text
进入回放界面
查看回放列表
选择回放记录
播放回放
查看车辆数据

```

这条分支也不需要进入实时小车控制流程，它读取的是之前保存的回放数据。

## 5. 后台服务 / Redis 黑板泳道怎么画

你之前问过 FastAPI/调度器要不要写。这里建议不要直接写得太技术化，图中标题写：

```text
后台服务 / Redis 黑板
```

这样老师更容易理解。

这个泳道负责三类事情：

### 5.1 登录与权限

```text
接收登录请求
校验账号密码
生成 token
按角色返回界面
校验接口权限
```

权限规则来自 `backend/app/main.py`：

| 路径 | 允许角色 |
| --- | --- |
| `/admin/*` | 超级管理员 |
| `/control/*`、`/runtime/*`、`/state`、`/ws` | 系统运行员 |
| `/replays` | 超级管理员、分析员 |

图中可以画一个判断菱形：

```text
角色权限匹配？
```

否的话：

```text
返回无权限提示
```

### 5.2 黑板状态

Redis 黑板保存这些核心数据：

```text
map
vehicles
frontiers
tasks
navigationRequests
navigationPlans
heartbeats
events
replays
auth users
```

图中可以把 Redis 黑板画成一个绿色数据块或数据库圆柱，放在“后台服务 / Redis 黑板”泳道中部。

### 5.3 快照推送

```text
生成状态快照
通过 WebSocket 推送
前端刷新 3D 地图和面板
```

这个流程应该从 Redis 黑板回到界面泳道。

## 6. 控制器、小车、导航器主流程

这部分是系统运行员点击“开始探索”之后发生的。

### 6.1 小车泳道

建议节点：

```text
开始小车
初始化小车
写入心跳
等待任务
获取小车状态
状态为空？
状态为 IDLE？
获取任务队列
任务为空？
执行路径
扫描周围地图
上传 map patch
发现 frontier
状态改为 IDLE / MOVING
遇到障碍则上报阻塞
```

和参考图对应关系：

- “等待书拍”可以换成“等待任务”。
- “获取任务队列”保留。
- “状态为空？”可以对应你项目里的车辆状态是否存在。
- “状态为1？”可以改成“车辆是否空闲？”。
- “状态改为2”可以改成“状态改为 MOVING”。
- “状态改为0，删除任务队列”可以改成“任务完成，车辆回到 IDLE”。

### 6.2 控制器泳道

建议节点：

```text
开始控制器
循环读取 Redis 黑板
读取 OPEN frontier
读取空闲小车
队列长度是否为空？
是否有可分配 frontier？
执行任务分配策略
创建任务 task
创建导航请求 navigationRequest
写入 Redis 黑板
```

任务分配策略：

```text
nearest
greedy
low_mdp 特殊模式
```

注意：

```text
low_mdp 模式下，跳过 Controller 和 Navigator，
小车直接基于局部 MDP 选择下一步。
```

### 6.3 导航器泳道

建议节点：

```text
开始导航器
循环监听导航请求
收到 navigationRequest？
初始化地图信息和算法选择
等待工作监听 navigationRequests 队列
领取导航任务
选择 baseline / cbs
规划路径
找到合适终点和路径？
写入 navigationPlan
路径写入任务 pathQueue
```

算法分支：

```text
baseline -> A* / 时空 A*
cbs -> 多车路径冲突检测和约束重规划
```

失败分支：

```text
找不到路径
  -> 写入 NO_PATH
  -> 记录系统事件
  -> 等待下一轮重规划
```

## 12. 图上不建议写太细的内容

不要把这些东西全部塞进图里：

```text
完整接口地址
每个函数名
A* 公式
CBS 约束细节
算法实现中的低层参数细节
Three.js 模型资源路径
Redis key 的完整前缀
```

这些放在报告文字或答辩说明里即可。多泳道图只保留“谁做什么、数据怎么流、状态怎么反馈”。

## 13. 答辩讲解词

可以这样讲：

```text
这张多泳道图先从界面登录开始。用户登录后，系统根据角色分为超级管理员、系统运行员和分析员三条分支。超级管理员进入用户管理界面，负责新增、编辑、删除系统运行员和分析员账号；分析员进入回放分析界面，查看历史运行记录、播放回放并查看每一帧对应的车辆和黑板数据；系统运行员进入仿真运行界面，配置地图、障碍、小车数量、任务分配策略和路径规划算法，然后点击开始探索。

系统运行后，后台服务负责校验权限并把运行配置写入 Redis 黑板。Redis 黑板保存 map、vehicles、frontiers、tasks、navigationRequests、navigationPlans、heartbeats、events 和回放数据。小车先扫描地图并上传 map patch，发现 frontier 后写入黑板；控制器读取开放 frontier 和空闲小车，根据 nearest 或 greedy 策略创建任务和导航请求；导航器领取导航请求，使用 A*、时空 A* 或 CBS 规划路径，并把路径计划写回黑板；小车再读取路径队列执行移动。前端通过 WebSocket 接收状态快照，所以可以实时看到地图、车辆、任务、路径和事件变化。
```

最后强调：

```text
这个系统的核心特点是角色权限清晰，运行流程完整，并且 Robot、Controller、Navigator 不直接互相调用，而是通过 Redis 黑板交换状态，因此系统耦合度低，后续容易扩展成分布式 worker。
```

## 14. 最终绘图检查清单

画完后检查：

- 是否有“界面 / 后台服务与 Redis 黑板 / 控制器 / 小车 / 导航器”泳道。
- 界面泳道是否画出了超级管理员、系统运行员、分析员三个角色。
- 超级管理员是否只走用户管理流程。
- 分析员是否只走回放查看和数据分析流程。
- 系统运行员是否进入地图配置、车辆配置、算法选择和启动仿真流程。
- Redis 黑板是否展示了 map、vehicles、frontiers、tasks、navigationRequests、navigationPlans、events、replays。
- 控制器、小车、导航器是否形成任务分配、路径规划、执行反馈闭环。
- 是否有 WebSocket 快照回到前端实时显示。
- 图中是否避免塞入过多代码细节。
