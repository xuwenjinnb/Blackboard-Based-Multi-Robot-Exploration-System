# 后端小车、导航器、控制器算法讲解（含论文引用）

这份文档用于给老师演示时讲解。核心说法可以先记住一句：

> 后端把多车探索拆成三层：小车负责“感知和执行”，控制器负责“任务分配”，导航器负责“路径规划和避让冲突”。三层通过黑板 Blackboard 交换地图、车辆、frontier、任务和路径计划。

代码范围：

- 控制器：`backend/app/controller.py:24`，`backend/app/policies.py:56`，`backend/app/policies.py:128`
- 导航器：`backend/app/navigator/component.py:20`，`backend/app/pathfinding.py:44`，`backend/app/pathfinding.py:105`，`backend/app/navigator/planners/cbs_planner.py:17`
- 小车：`backend/app/robot/component.py:49`，`backend/app/robot/algorithms/frontier_scan.py:32`，`backend/app/robot/algorithms/frontier_scan.py:72`，`backend/app/robot/algorithms/low_mdp.py:75`

## 1. 总体运行流程

普通模式下，每个仿真 tick 的流程是：

```text
小车扫描局部地图
  -> 发现 frontier 边界点
  -> 控制器给空闲小车分配 frontier
  -> 导航器规划到 frontier 的路径
  -> 小车按路径移动
  -> 移动后继续扫描
```

对应代码在 `SimulationEngine._step()`：

- 如果不是 `low_mdp`，顺序执行 `robot_phase()`、`controller_phase()`、`navigator_phase()`。
- 如果是 `low_mdp`，直接进入 `low_level_mdp_phase()`，小车自己做局部决策，控制器和导航器不再参与目标分配与全局规划。

演示时可以这样讲：

> 我们这里不是把所有逻辑写成一个大函数，而是模拟分布式机器人系统。小车像执行器和传感器，控制器像任务调度中心，导航器像路径规划服务。它们都读写黑板，所以可以很自然地扩展成多个小车、多个导航器并行运行。

## 3. 控制器算法：决定“哪辆车去哪”

入口：`ControllerComponent.run_once()`。

控制器不直接控制小车移动，也不算具体路径。它做的是任务分配：

1. 从黑板读取所有 `OPEN` frontier。
2. 从黑板读取所有 `IDLE` 空闲车辆。
3. 调用分配策略 `select_assignments()`。
4. 给每个分配结果创建 task。
5. 给 task 创建 navigation request，交给导航器规划路径。

### 3.1 Nearest Reachable Frontier

代码：`NearestReachableFrontierPolicy`，位置 `backend/app/policies.py:128`。

核心逻辑：

```text
对每辆空闲车：
  遍历候选 frontier
  跳过没有未知收益的 frontier
  用 A* 判断是否可达
  选择真实路径距离最短的 frontier
```

这里的“最近”不是直线最近，而是 A* 找出来的路径最短。这样如果中间有障碍，控制器不会误以为一个看起来很近但绕不过去的点更好。

可以对老师这样讲：

> nearest 策略的重点是“可达”。它先用 frontier 思想找探索边界，再用 A* 验证车辆真的能到达，最后按路径长度选最近目标。这比只看欧氏距离或曼哈顿距离更可靠。

论文对应：

- Frontier 的概念来自 Yamauchi 的 frontier-based exploration。
- 多机器人目标分配中考虑“目标收益”和“到达代价”的思想，对应 Burgard 等人的 coordinated multi-robot exploration。

### 3.2 Greedy Information Gain

代码：`GreedyFrontierPolicy`，位置 `backend/app/policies.py:56`。

打分公式：

```text
assignment_score = unknown_gain * gain_weight - distance_weight * manhattan_distance
```

默认参数：

```text
gain_weight = 10.0
distance_weight = 0.35
```

含义：

- `unknown_gain` 越大，说明到这个 frontier 后能看到更多未知区域，收益越高。
- 距离越远，成本越高，要扣分。
- 最后选择分配分最高的 frontier。这里的 `assignment_score` 只是控制器内部临时计算的分数，不会作为 frontier 字段保存到 Redis。

可以这样讲：

> greedy 策略不是只贪最近，而是贪“信息收益”。如果一个 frontier 远一点，但是它背后未知区域很多，它仍然可能被选中。这个思想和多机器人探索论文里的 utility-cost 权衡是一致的。

### 3.3 low_mdp 下的 Noop 控制器

如果策略名是 `low_mdp`，`create_assignment_policy()` 会返回 `NoopAssignmentPolicy`。

它永远返回空分配：

```text
select_assignments(...) -> []
```

原因是 low-level MDP 模式下，小车自己决定下一步去哪，不需要控制器给它分配 frontier，也不需要导航器规划全局路径。

## 4. 导航器算法：决定“怎么走过去”

入口：`NavigatorComponent.run_once()`。

导航器领取控制器创建的 navigation request，然后写回 navigation plan。成功时路径会进入任务的 `pathQueue`，小车再按这个队列执行。

### 4.1 A* 单车路径规划

代码：`backend/app/pathfinding.py:44`。

状态：

```text
(x, y)
```

启发函数：

```text
h = |x - goal_x| + |y - goal_y|
```

代价设计：

- `OBSTACLE` / `RESERVED`：不可通行
- `UNKNOWN`：代价 3.0
- `FREE`：代价 1.0
- `VISITED`：代价 0.8

所以本项目的 A* 有一个工程偏好：更愿意走已知安全区域，必要时也可以穿过未知区域，只是代价更高。

可以这样讲：

> A* 的理论形式是 `f=g+h`。`g` 是已经走过的代价，`h` 是到目标的估计代价。我们这里用曼哈顿距离做启发，因为地图是四方向栅格。代码还给不同地图状态设置了不同代价，让路径更偏向安全的已知区域。

论文对应：

- Hart、Nilsson、Raphael 在 1968 年提出 A* 的形式化基础。

### 4.2 Space-Time A* 时空 A*

代码：`backend/app/pathfinding.py:105`，封装在 `StAStarPlanner`。

普通 A* 只搜索空间位置，时空 A* 把时间也放进状态：

```text
(x, y, t)
```

这样就可以表达“这个格子现在能走，但下一秒可能被别的车占用”。

本项目使用两类预约：

```text
reservations[t] = 某时间片已经被占用的格子
edge_reservations = 某时间片已经被占用的移动边
```

其中 edge reservation 用来避免两辆车对向交换位置。例如 A 从 `(1,1)` 去 `(1,2)`，B 同时从 `(1,2)` 去 `(1,1)`，虽然它们没有在同一时刻占同一格，但会在边上迎面相撞。

时空 A* 还允许 `WAIT`：

```text
neighbors = 上下左右 + 原地等待
```

可以这样讲：

> 时空 A* 相当于把地图从二维变成三维：横轴、纵轴和时间轴。小车不只是问“这个格子能不能走”，还要问“这个时间点能不能占用这个格子”。如果前方被预约了，它可以等待一拍。

论文对应：

- Silver 2005 的 Cooperative Pathfinding 提出了在 space-time 中搜索非碰撞路径的 Cooperative A* 思路，本项目的 `st_astar` 就是这个思想的工程实现。

### 4.3 CBS 冲突搜索

代码：`backend/app/navigator/planners/cbs_planner.py:17`。

CBS 用于多车联合规划。它和时空 A* 的区别是：

- 时空 A*：一个请求一个请求规划，避让已有预约。
- CBS：一次拿一批请求，先分别规划，再统一检测冲突，冲突后加约束重规划。

CBS 两层结构：

```text
高层：维护约束树，发现冲突后生成新约束
低层：在约束下给单辆车重新规划路径
```

本项目低层搜索调用的是 `st_astar`。

CBS 检测两类冲突：

```text
顶点冲突：同一时间，两辆车进入同一格
边冲突：同一时间，两辆车对向交换位置
```

发现冲突后，CBS 会生成两个子节点：

```text
子节点 1：限制车 A 不能在该时间进入该格/经过该边
子节点 2：限制车 B 不能在该时间进入该格/经过该边
```

然后只对被限制的那辆车重新规划，总代价小的解优先展开。

可以这样讲：

> CBS 的思路很像“先各走各的，再解决矛盾”。如果两辆车撞了，不是把所有车一起暴力搜索，而是只对冲突车辆加约束并重新规划。这样比把所有车辆合成一个超大状态空间要轻很多。

论文对应：

- Sharon、Stern、Felner、Sturtevant 的 Conflict-Based Search。原始 AAAI 2012 版本强调两层搜索；2015 年 Artificial Intelligence 期刊版给出了更完整的 MAPF 表述。

## 5. 小车算法：负责“执行、扫描、发现边界”

入口：`RobotComponent.run_vehicle_once()`。

小车有三类行为：

1. 没任务时：扫描周围环境，更新地图。
2. 有路径时：按 `pathQueue` 执行下一步。
3. 遇到真实障碍时：上报阻塞，清空旧路径，触发重规划。

### 5.1 路径执行

代码：`backend/app/robot/component.py:49`。

每轮执行：

```text
读取当前任务
  如果没有任务：扫描并保持 IDLE
  如果任务还没规划好：先扫描
  如果有 pathQueue：执行 currentStepIndex + 1
  如果下一步被真实障碍挡住：report_blocked，等待重规划
  如果可走：更新位置、朝向、电量、任务进度，并扫描
```

可以这样讲：

> 小车不自己猜全局路线，它相信导航器写入的 pathQueue。执行时如果发现真实障碍，说明地图信息落后于真实环境，于是它先扫描、上报阻塞，再让系统重新规划。

### 5.2 扫描与地图 patch

代码：`FrontierScanAlgorithm.scan_and_upload()`，位置 `backend/app/robot/algorithms/frontier_scan.py:32`。

扫描范围：

```text
以车辆当前位置为中心，边长 2 * scan_radius + 1 的正方形
```

默认 `scan_radius = 2`。

扫描规则：

- 真实障碍：标记为 `OBSTACLE`
- 小车所在格：标记为 `VISITED`
- 其他可见格：标记为 `FREE`

小车把这些格子打包成 map patch 上传给黑板。黑板合并 patch 后，地图版本更新。

### 5.3 Frontier 检测

代码：`FrontierScanAlgorithm.detect_frontiers()`，位置 `backend/app/robot/algorithms/frontier_scan.py:72`。

frontier 的判断条件：

```text
当前格必须是 FREE 或 VISITED
并且八邻域中存在 UNKNOWN
```

再计算探索收益：

```text
unknown_gain = 该位置扫描半径内可见 UNKNOWN 格子数
rank_score = unknown_gain * 10.0 - 0.15 * distance
```

这里的 `rank_score` 只用于本次扫描时给候选 frontier 排序，不会写入 Redis；Redis 里的 frontier 只保留 `unknownGain`。

可以这样讲：

> frontier 就是已知区域和未知区域的边界。小车每次扫描后，会在附近找这种边界点，并把它们作为候选探索目标。这样探索就不是随机乱走，而是不断把已知地图边界向外推进。

论文对应：

- Yamauchi 1997 明确提出 frontier 是开放空间和未知空间之间的边界，机器人移动到 frontier 后可以扩展地图。
- Yamauchi 1998 把这个思想扩展到多机器人场景，机器人共享感知信息，但可以独立选择探索目标。本项目通过黑板共享地图和 frontier。

## 6. Low-Level MDP 小车自主决策

代码：`LowLevelMDPAlgorithm.choose_step()`，位置 `backend/app/robot/algorithms/low_mdp.py:75`。

当策略是 `low_mdp` 时，小车不再等待控制器分配目标，也不等待导航器规划路径，而是在局部已知区域中自己选择下一步。

这一块一定要讲清楚：**主算法归属是 2012 年 Distributed Value Functions / Dec-MDP，不是 2023 年 MADE-Net 本体算法。**

准确对应关系是：

```text
主算法思想：Distributed Value Functions for Multi-Robot Exploration
对应年份：2012
最新文章关系：2023 MADE-Net 论文把这类 MDP/DVF 方法作为经典对比背景/基线
理论基础：MDP / 动态规划 / 价值迭代
```

本项目的 `low_mdp` 更接近“分布式价值函数 / 去中心化 MDP 探索”的传统算法：每辆车维护局部可达状态，对未知区域给正奖励，对靠近其他车的位置给负惩罚，通过价值传播选择下一步。它不是 MADE-Net 那篇文章中的深度强化学习网络本体，因为代码里没有神经网络、经验回放、Q 网络或宏动作策略训练；更准确地说，它是 **2012 年 DVF/Dec-MDP 类算法的工程化简化实现**。2023 年 MADE-Net 论文可以用来说明“最新去中心化多机器人探索研究会拿这类 MDP 方法做对比”，但不能说本项目实现了 MADE-Net。

局部状态空间：

```text
从当前位置 BFS
只保留 horizon 范围内的 FREE / VISITED 格子
```

默认参数：

```text
horizon = 20
discount = 0.80
iterations = 50
repulsion_weight = 8.0
move_cost = 0.2
```

奖励函数：

```text
reward(s) = 从状态 s 扫描能看到的 UNKNOWN 格子数
```

斥力惩罚：

```text
penalty += weight * discount ^ distance
```

作用是让小车远离其他车，减少扎堆。

价值迭代形式：

```text
V(s) = reward(s) - repulsion(s) + discount * max(V(s') - move_cost)
```

最后只在当前位置的上下左右和原地等待中选下一步：

```text
score = reward(candidate)
      - repulsion(candidate)
      + discount * V(candidate)
      - move_penalty
```

可以这样讲：

> low-level MDP 模式对应 2012 年左右去中心化多机器人探索中的 Distributed Value Functions / Dec-MDP 思想。每辆车不等待中心控制器给目标，而是在自己的局部状态空间里计算价值：能看到更多未知区域的格子价值更高，靠近其他车或被其他车选中的位置价值降低。这样每辆车都能局部决策，同时又通过斥力/分布式价值项减少重复探索。2023 年 MADE-Net 是更新的深度强化学习方法，但我们这里没有实现它的神经网络和宏动作学习，只能把它作为相关最新研究背景来提。

注意：

- 这不是完整随机 MDP，而是工程上的确定性局部 MDP。
- 它主要借鉴 Dec-MDP / Distributed Value Functions 的多车分布式探索思想。
- Bellman 动态规划和价值迭代是它的底层数学基础，但演示时不应该只引用 Bellman。
- 它适合展示“没有中心控制器时，小车也能靠局部收益继续探索”。

## 7. 演示时的 1 分钟讲稿

可以直接照着说：

> 我们后端采用黑板架构，把多车探索拆成小车、控制器和导航器三个组件。小车负责扫描地图、上传局部 patch，并检测 frontier，也就是已知可通行区域和未知区域的边界。这个思想对应 Yamauchi 的 frontier-based exploration。控制器负责把 frontier 分配给空闲小车，支持最近可达策略和信息增益贪心策略，后者用未知区域收益减去距离成本，和多机器人探索里的 cost-utility 思想一致。导航器负责路径规划，普通情况下用 A*，如果要避让车辆就用时空 A*，把状态扩展成 `(x,y,t)` 并用预约表避免碰撞；如果配置 CBS，就批量处理多车请求，通过冲突搜索不断加约束来消除顶点冲突和边冲突。最后小车按路径执行，遇到真实障碍会扫描并触发重规划。如果切到 low_mdp 模式，小车就不依赖控制器和导航器，而是在局部状态空间里用分布式价值函数思想自主选择下一步。这个 low-mdp 主体对应 Matignon 等人 2012 年的 Distributed Value Functions/Dec-MDP 探索方法，不是 Tan 等人 2023 年 MADE-Net 的深度强化学习网络。

## 8. 老师可能会问的问题

**Q1：为什么不直接让小车随机走？**

随机走效率低，容易重复。frontier 方法会主动朝“未知区域边界”移动，能更稳定地扩大地图。

**Q2：nearest 和 greedy 有什么区别？**

nearest 更重视路径短和可达性；greedy 更重视信息收益，可能为了看到更多未知区域选择更远的 frontier。

**Q3：A* 和时空 A* 的区别是什么？**

A* 只考虑空间位置 `(x,y)`；时空 A* 考虑 `(x,y,t)`，能避免未来时间片上的车辆占用冲突。

**Q4：CBS 相比时空 A* 有什么优势？**

时空 A* 是顺序避让已有路径；CBS 是对一批车联合检查冲突，并通过约束树重规划，更适合多车同时规划。

**Q5：low_mdp 为什么不用导航器？**

因为 low_mdp 的目标是展示局部自主决策。小车每轮自己计算下一步，而不是先拿全局目标再规划完整路径。

**Q6：low_mdp 是不是最新那篇文章的算法？**

不是。演示时不要说“我们实现的是 MADE-Net”。更准确的话术是：“low-mdp 是基于 Distributed Value Functions / Dec-MDP 的去中心化 MDP 探索基线，主要对应 2012 年 Matignon 等人的算法思想；2023 年 MADE-Net 是更新的深度强化学习方法，它可以作为相关最新论文背景，但本项目没有实现 MADE-Net 网络。”
