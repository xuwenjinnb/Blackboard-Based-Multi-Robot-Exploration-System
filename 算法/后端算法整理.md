# 后端控制器、导航器与小车算法整理

本文整理路径：

- 后端代码目录：`backend/app`
- 控制器：`controller.py`、`controller/policies/*`
- 导航器：`navigator/component.py`、`navigator/planners/*`
- 小车：`robot/component.py`、`robot/algorithms/*`
- 调度入口：`simulation.py`

## 1. 总体运行流程

后端采用黑板架构。`Blackboard` 保存地图、车辆、边界点、任务、导航请求、路径规划结果和事件日志。`SimulationEngine` 每个仿真 tick 会调度三个主要组件：

1. 小车组件 `RobotComponent`
   - 执行已有路径。
   - 扫描周围环境。
   - 上传地图 patch。
   - 发现新的 frontier 边界点。

2. 控制器组件 `ControllerComponent`
   - 查看未分配的 frontier。
   - 查看空闲车辆。
   - 按策略选择“哪辆车去哪个 frontier”。
   - 创建任务和导航请求。

3. 导航器组件 `NavigatorComponent`
   - 领取导航请求。
   - 调用路径规划算法。
   - 将成功或失败的路径计划写回黑板。

默认流程是：

```text
小车扫描地图 -> 发现 frontier -> 控制器分配 frontier -> 导航器规划路径 -> 小车执行路径 -> 继续扫描
```

如果启用 `low_mdp` 策略，则流程会变成：

```text
小车直接用低层 MDP 选择下一步 -> 移动/等待 -> 扫描地图 -> 直到未知区域全部探索完成
```

此时控制器不会再给小车分配 frontier，导航器也不再负责规划路径。

## 2. 控制器算法

控制器入口是 `ControllerComponent.run_once()`。

控制器本身不直接做路径规划，它负责“任务分配”。它从黑板读取当前地图、车辆和 frontier，然后调用不同的 `AssignmentPolicy` 生成分配决策。

### 2.1 控制器每轮做的事情

1. 更新心跳状态为 `BUSY`。
2. 对已有但还没有导航请求的 `PENDING` 任务补发导航请求。
3. 读取当前地图快照 `snapshot`。
4. 读取所有 `OPEN` 状态的 frontier。
5. 读取所有空闲车辆。
6. 调用分配策略 `select_assignments()`。
7. 为每个分配结果创建任务：
   - 任务状态为 `PENDING`。
   - frontier 状态改为 `ASSIGNED`。
   - 车辆记录当前任务 ID。
8. 为任务创建导航请求。
9. 更新心跳状态为 `READY`。

### 2.2 Nearest Reachable Frontier 最近可达边界策略

代码位置：`policies.py` 中的 `NearestReachableFrontierPolicy`

这是默认策略，对应前端/接口里的 `nearest`。

核心思想：

> 对每辆空闲车，在所有可用 frontier 中找一条真实可达路径，然后选择路径距离最短的 frontier。

具体步骤：

1. 收集已被车辆占用的位置和已有任务目标位置。
2. 对每辆空闲车，遍历当前可用 frontier。
3. 跳过已经被占用的 frontier 位置。
4. 计算该 frontier 周围是否还能看见未知格子：
   - 如果 `unknown_cells_visible_from(...) <= 0`，说明这个 frontier 没有探索收益，跳过。
5. 调用 A* 判断从车辆当前位置到 frontier 是否可达。
6. 如果 A* 找不到路径，跳过该 frontier。
7. 记录路径长度 `distance` 和曼哈顿距离 `manhattan`。
8. 按 `(distance, manhattan)` 升序排序。
9. 选择路径距离最短的 frontier。
10. 每个 frontier 只分配给一辆车。

这个策略的优点是稳定、容易理解，并且会用真实路径距离而不是单纯直线距离来判断目标远近。

### 2.3 Greedy Information Gain 信息增益贪心策略

代码位置：`policies.py` 中的 `GreedyFrontierPolicy`

对应前端/接口里的 `greedy`。

核心思想：

> frontier 周围未知格子越多越好，距离越远越差，综合打分后选最高分。

评分公式：

```text
assignment_score = unknown_gain * gain_weight - distance_weight * manhattan_distance
```

默认参数：

- `gain_weight = 10.0`
- `distance_weight = 0.35`

具体步骤：

1. 对每辆空闲车遍历所有可用 frontier。
2. 计算 frontier 周围扫描半径内的未知格子数量 `unknown_gain`。
3. 如果 `unknown_gain <= 0`，说明没有探索收益，跳过。
4. 计算车辆当前位置到 frontier 的曼哈顿距离。
5. 使用公式计算得分。
6. 选择得分最高的 frontier。
7. 被选中的 frontier 从可用列表中删除，避免重复分配。

这个策略更偏向探索收益：即使目标远一点，只要未知区域足够多，也可能优先选择。

### 2.4 Noop 策略

代码位置：`policies.py` 中的 `NoopAssignmentPolicy`

当策略名是 `low_mdp` 时，控制器使用 Noop 策略。

它的行为是：

```text
select_assignments(...) -> []
```

也就是不产生任何分配结果。因为 `low_mdp` 模式下，小车自己直接决策下一步，不需要控制器分配 frontier，也不需要导航器规划全局路径。

## 3. 导航器算法

导航器入口是 `NavigatorComponent.run_once()`。

导航器负责把控制器创建的导航请求转换成路径计划。路径计划写回黑板后，小车才能按 `pathQueue` 执行。

### 3.1 导航器每轮做的事情

普通模式：

1. 每个 navigator ID 尝试领取一个 `PENDING` 导航请求。
2. 如果领取成功，请求状态改为 `PLANNING`。
3. 根据配置和请求选择规划器。
4. 调用规划器计算路径。
5. 如果成功，写入 `SUCCESS` 计划。
6. 如果失败，写入 `NO_PATH` 计划和失败原因。

CBS 模式：

1. 使用第一个 navigator 一次性领取多个 `PENDING` 请求。
2. 把这些请求作为一个批次交给 CBS 规划器。
3. CBS 对多辆车的路径统一检测冲突并加约束重规划。
4. 对每个请求分别写回路径计划。

### 3.2 Baseline A* 单车路径规划

代码位置：

- `navigator/planners/astar_planner.py`
- `navigator/planners/astar_planner.py` 中的 `astar`

普通 A* 用来为单个请求规划从 `start` 到 `goal` 的路径。

搜索状态：

```text
(x, y)
```

邻居扩展：

```text
上、下、左、右四方向
```

启发函数：

```text
h = 曼哈顿距离
```

格子代价：

- `OBSTACLE` / `RESERVED`：不可走。
- `UNKNOWN`：代价 3.0。
- `FREE`：代价 1.0。
- `VISITED`：代价 0.8。

这意味着算法更愿意走已经访问过或确定空闲的区域，但在必要时也可以穿过未知区域，只是代价更高。

输出路径包含：

- `stepIndex`
- `position`
- `heading`
- `expectedTimeSlot`
- `action = MOVE`

失败情况包括：

- 起点或终点在地图外。
- 终点是障碍或保留格。
- 搜索结束仍找不到路径。

### 3.3 Space-Time A* 时空 A*

代码位置：

- `navigator/planners/st_astar_planner.py`
- `navigator/planners/st_astar_planner.py` 中的 `st_astar`

当请求要求避让车辆，并且配置 `use_st_astar = True` 时，导航器会使用时空 A*。

搜索状态：

```text
(x, y, t)
```

与普通 A* 相比，时空 A* 多了时间维度。它不仅判断某个格子能不能走，还判断某个时间点能不能占用该格子。

额外约束：

1. `reservations`
   - 记录某个时间片已经被其他车辆占用的格子。
2. `edge_reservations`
   - 记录某个时间片已经被其他车辆使用的边。
   - 用来避免两辆车对向交换位置。
3. `WAIT` 动作
   - 时空 A* 的邻居包含原地等待。
   - 当下一时间片目标格子被占用时，小车可以选择等一拍。

预约表构造方式：

1. 遍历其他未完成任务。
2. 读取它们的 `pathQueue` 和当前执行进度。
3. 把未来路径按时间片加入 `reservations`。
4. 把连续两个路径点形成的边加入 `edge_reservations`。
5. 对已经到达路径末端的车辆，将最终位置一直保留到规划时域结束。
6. 对没有活动任务的车辆，把当前位置在 `t = 0` 加入预约。

默认规划时域：

```text
st_astar_horizon = 96
```

输出路径里可能包含：

```text
action = MOVE
action = WAIT
```

这个算法适合处理“已有车辆路径已知，新请求需要避让它们”的情况。

### 3.4 CBS 冲突搜索算法

代码位置：`navigator/planners/cbs_planner.py`

CBS 全称 Conflict-Based Search，用来做多车联合路径规划。

与时空 A* 的区别：

- 时空 A* 是一个请求一个请求规划，避让已有预约。
- CBS 是批量领取多个请求，先各自规划，再统一检测冲突并加约束重规划。

CBS 的整体流程：

1. 批量领取多个导航请求。
2. 构造固定预约：
   - 已经在执行、但不属于本批次的任务路径。
   - 没有参与本批次规划的车辆当前位置。
3. 对每个请求调用低层搜索。
   - 低层搜索本质上还是 `st_astar`。
4. 得到初始解后，把它放入 CBS 的优先队列。
5. 优先队列按总代价排序：

```text
solution_cost = 所有车辆路径长度之和
```

6. 每次弹出总代价最低的解。
7. 检测路径冲突。
8. 如果没有冲突，返回成功。
9. 如果有冲突，分别对冲突中的两辆车加约束，生成两个子节点。
10. 对加了约束的车辆重新调用低层搜索。
11. 新解继续放回优先队列。
12. 超过最大节点数仍无法解决，则返回失败。

CBS 检测两类冲突：

1. 顶点冲突

```text
同一时间片，两辆车占用同一个格子
```

2. 边冲突

```text
同一时间片，两辆车互相交换位置
```

CBS 生成两类约束：

1. 顶点约束

```text
某辆车在某个时间片不能进入某个格子
```

2. 边约束

```text
某辆车在某个时间片不能经过某条边
```

默认最大搜索节点数：

```text
cbs_max_nodes = 128
```

CBS 更适合多车同时产生导航请求、并且路径之间容易冲突的场景。

### 3.5 导航器输出的路径计划

成功时写回：

- `status = SUCCESS`
- `planner`
- `path`
- `cost = len(path)`
- `distance = len(path) - 1`
- `estimatedTime = len(path) - 1`
- `mapVersion`

失败时写回：

- `status = NO_PATH`
- `path = []`
- `failReason`
- `planner`

黑板收到成功计划后，会把任务状态改为 `PLANNED`，并把路径放进任务的 `pathQueue`。

## 4. 小车算法

小车入口是 `RobotComponent.run_once()`。

小车负责执行路径、更新车辆状态、扫描地图、发现 frontier。如果启用 `low_mdp`，小车还会自己决定下一步移动方向。

### 4.1 普通路径执行算法

代码位置：`robot/component.py`

当没有启用 `low_mdp` 时，小车按导航器写入的路径执行。

每轮逻辑：

1. 遍历所有车辆。
2. 查询车辆当前任务。
3. 如果车辆没有任务：
   - 状态设为 `SCANNING`。
   - 扫描并上传地图。
   - 状态设回 `IDLE`。
4. 如果任务状态是 `PENDING` 或 `FAILED`：
   - 说明还没有可执行路径，先扫描环境。
5. 如果任务有 `pathQueue`：
   - 将任务标记为 `RUNNING`。
   - 当前步 `currentStepIndex + 1` 作为下一步。
6. 如果已经走到路径末尾：
   - 任务标记为 `DONE`。
   - frontier 标记为 `VISITED`。
   - 车辆变为 `IDLE`。
7. 如果下一步在真实障碍中：
   - 先扫描环境。
   - 上报阻塞。
   - 任务重置为 `PENDING`。
   - 清空旧路径并增加 `replanCount`。
   - 等待控制器重新发起导航请求。
8. 如果下一步可走：
   - 更新车辆位置、朝向、状态、任务 ID、计划 ID。
   - 电量减 1。
   - 更新任务进度。
   - 移动后扫描并上传地图。

这个算法让小车始终以黑板中的任务和路径为准，遇到真实障碍会触发重规划。

### 4.2 扫描与地图上传算法

代码位置：`robot/algorithms/frontier_scan.py`

扫描算法由 `FrontierScanAlgorithm.scan_and_upload()` 实现。

扫描范围：

```text
以车辆当前位置为中心，边长为 2 * scan_radius + 1 的正方形区域
```

默认：

```text
scan_radius = 2
```

扫描每个格子的规则：

1. 如果格子越界，跳过。
2. 如果真实地图中该格子是障碍，标记为 `OBSTACLE`。
3. 如果是车辆当前所在格子，标记为 `VISITED`。
4. 其他可见格子标记为 `FREE`。

之后小车会把这些格子打包成地图 patch 上传到黑板：

- `patchId`
- `vehicleId`
- `baseMapVersion`
- `cells`
- `timestamp`

黑板合并 patch 后，地图版本号会更新。

### 4.3 Frontier 边界点检测算法

代码位置：`robot/algorithms/frontier_scan.py`

frontier 的含义：

> 已知可通行区域和未知区域的边界点。它本身必须是 `FREE` 或 `VISITED`，同时邻域中存在 `UNKNOWN`。

检测范围：

```text
以车辆当前位置为中心，半径 frontier_search_radius 的正方形区域
```

默认：

```text
frontier_search_radius = 5
```

检测步骤：

1. 读取当前地图。
2. 遍历车辆附近的格子。
3. 只保留 `FREE` 或 `VISITED` 格子。
4. 判断八邻域中是否存在 `UNKNOWN`。
5. 计算该点在扫描半径内能看到多少未知格子：

```text
unknown_gain = unknown_cells_visible_from(position)
```

6. 如果 `unknown_gain <= 0`，跳过。
7. 计算距离惩罚：

```text
distance_penalty = 0.15 * manhattan(center, position)
```

8. 计算 frontier 分数：

```text
rank_score = unknown_gain * 10.0 - distance_penalty
```

9. 按 `(rank_score, unknown_gain)` 降序排序。`rank_score` 只在本次扫描内部排序使用，不写入 Redis。
10. 保存 frontier 到黑板。

黑板保存 frontier 时会按位置去重：同一个位置重复上传不会生成多个 frontier。

### 4.4 Low-Level MDP 小车低层决策算法

代码位置：`robot/algorithms/low_mdp.py`

对应策略名：

```text
low_mdp
low-mdp
low-level-mdp
low_level_mdp
```

核心思想：

> 不让控制器分配目标，也不让导航器规划全局路径，而是每辆小车在局部已知区域内做价值迭代，直接选择下一步。

启用条件：

```text
config.policy in {"low_mdp", "low-mdp", "low-level-mdp", "low_level_mdp"}
```

低层 MDP 每轮流程：

1. 初始化本轮已经选择的下一位置 `selected_next`。
2. 初始化本轮已经选择的移动边 `selected_edges`。
3. 逐辆车读取最新快照。
4. 调用 `choose_step()` 选择下一格。
5. 如果没有可走动作，则原地等待。
6. 记录本车选择的下一位置和移动边。
7. 更新车辆位置、朝向、状态和电量。
8. 移动后扫描并上传地图。
9. 所有车辆执行完后，统一把状态设为 `IDLE`。

局部状态空间构造：

`local_known_traversable_cells()` 从车辆当前位置开始 BFS，只收集：

- 距离不超过 `low_level_mdp_horizon` 的格子。
- 起点允许保留。
- 其他格子必须是 `FREE` 或 `VISITED`。

默认参数：

- `low_level_mdp_horizon = 20`
- `low_level_mdp_discount = 0.80`
- `low_level_mdp_iterations = 50`
- `low_level_mdp_repulsion_weight = 8.0`
- `low_level_mdp_move_cost = 0.2`

奖励函数：

```text
reward(state) = 该位置扫描半径内可见 UNKNOWN 格子的数量
```

斥力场：

为了减少车辆扎堆，算法会对靠近其他车辆或本轮其他车辆已选择位置的状态施加惩罚：

```text
penalty += weight * discount ^ distance
```

其中：

- 其他车辆位置使用 `low_level_mdp_repulsion_weight`。
- 本轮已被其他车辆选中的下一位置使用两倍权重。
- 距离越远，惩罚越小。

价值迭代更新公式：

```text
V(s) = reward(s) - repulsion(s) + discount * max(V(s') - move_cost)
```

其中 `s'` 是四邻域加原地等待动作中的可行状态。

停止条件：

1. 达到最大迭代次数。
2. 或本轮最大价值变化小于 `0.01`。

选择下一步：

1. 只在当前位置的四邻域和原地等待中选。
2. 候选必须在局部状态空间中。
3. 不能移动到本轮其他车辆已经选择的下一格。
4. 不能产生对向交换边冲突。
5. 不能移动到真实障碍。
6. 计算动作分数：

```text
score = reward(candidate)
      - repulsion(candidate)
      + discount * V(candidate)
      - move_penalty
```

7. 按 `(score, 是否移动, reward)` 降序排序。
8. 选择分数最高的动作。

如果所有候选都没有正收益，算法会调用 `unknown_fallback_step()`：

1. 从当前位置 BFS。
2. 在已知可通行区域中寻找一个能看到未知格子的格子。
3. 返回通向该格子的第一步。
4. 如果找不到，则原地等待。

这个算法适合做“去中心化、低层自主探索”的实验。它的重点不是全局最短路，而是每辆车基于局部收益和多车斥力持续推进探索。

## 5. 三类算法的职责对比

| 模块 | 主要问题 | 输入 | 输出 | 典型算法 |
| --- | --- | --- | --- | --- |
| 控制器 | 哪辆车去哪个目标 | 空闲车辆、OPEN frontier、地图快照 | 任务、导航请求 | 最近可达 frontier、信息增益贪心 |
| 导航器 | 怎么从起点走到目标 | 导航请求、地图、车辆预约 | 路径计划 | A*、时空 A*、CBS |
| 小车 | 如何执行和感知 | 当前任务、路径、真实障碍、车辆状态 | 车辆新状态、地图 patch、frontier | 路径执行、局部扫描、frontier 检测、低层 MDP |

## 6. 关键配置项

代码位置：`config.py`

| 配置项 | 默认值 | 作用 |
| --- | --- | --- |
| `policy` | `nearest` | 控制器或小车使用的策略 |
| `navigator_algorithm` | `baseline` | 导航器模式，`baseline` 或 `cbs` |
| `scan_radius` | `2` | 小车扫描半径 |
| `frontier_search_radius` | `5` | 小车检测 frontier 的范围 |

| `st_astar_horizon` | `96` | 时空 A* 和 CBS 的时间规划上限 |
| `cbs_max_nodes` | `128` | CBS 最多扩展节点数 |
| `low_level_mdp_horizon` | `20` | 低层 MDP 的局部搜索范围 |
| `low_level_mdp_discount` | `0.80` | 低层 MDP 折扣因子 |
| `low_level_mdp_iterations` | `50` | 低层 MDP 价值迭代次数 |

## 7. 算法之间的数据流

普通模式数据流：

```text
Robot 扫描
  -> Blackboard.map 更新
  -> Robot 检测并保存 frontier
  -> Controller 读取 OPEN frontier 和 IDLE 车辆
  -> Controller 创建 task 和 navigation request
  -> Navigator 领取 request
  -> Navigator 规划 path
  -> Blackboard 写入 plan，并把 path 放入 task.pathQueue
  -> Robot 按 pathQueue 执行
```

低层 MDP 模式数据流：

```text
Robot 读取地图和车辆状态
  -> 构造局部状态空间
  -> 计算未知格子奖励和多车斥力
  -> 价值迭代选择下一步
  -> Robot 移动或等待
  -> 扫描并上传地图
  -> 地图无 UNKNOWN 后暂停仿真
```

## 8. 简短总结

控制器算法解决的是“目标分配”问题，决定每辆空闲车应该探索哪个 frontier。

导航器算法解决的是“路径规划”问题，决定车辆如何从当前位置到达目标，并处理障碍、车辆预约和多车冲突。

小车算法解决的是“执行与感知”问题，负责真实移动、扫描地图、发现 frontier；在 `low_mdp` 模式下，小车还会直接承担局部决策职责，不再依赖控制器和导航器。
