# 多小车协作巡检仿真项目验收展示流程

这份文档用于下周给老师验收时照着准备和展示。建议现场主线是：先把系统跑起来，再演示前端功能、实时仿真、算法切换和黑板低耦合架构，最后用设计文档回答扩展问题。

## 1. 项目一句话介绍

本项目是一个“多小车协作巡检仿真平台”。系统通过 FastAPI 后端、Redis 黑板和 3D 前端界面，让多辆小车在未知地图中协作探索、发现前沿点、分配任务、规划路径并实时可视化运行状态。

验收时可以这样开场：

```text
老师您好，我这个项目做的是多小车协作巡检仿真系统。核心目标是让多辆小车在未知地图中协作探索，并用 Redis 黑板把 Robot、Controller、Navigator 等组件解耦。前端可以配置地图、障碍、车辆数量、任务分配策略和路径规划算法，并通过 WebSocket 实时看到小车探索、任务生成、路径规划和系统事件。
```

### 4.1 常规启动

打开 PowerShell，进入后端目录：

```powershell
cd "C:\Users\Administrator\Desktop\Online continual learning\MOSE-master\3d\软件构造\实验\最后一次作业\backend"
```


```powershell
python -m venv .venv
```


## 5. 推荐现场展示顺序

总时长建议控制在 8 到 12 分钟。老师如果只给 5 分钟，就优先展示第 1 到第 5 步。

### 第 1 步：项目总体介绍，约 1 分钟

操作：

1. 打开 `http://127.0.0.1:8000/dashboard`。
2. 使用 `operator` / `123456` 登录运行控制台。
3. 先不急着点启动，展示页面整体布局。

要讲：

```text
这个页面是系统的运行控制台。上方可以选择任务分配策略和路径规划算法，中间是 3D 地图，右侧和下方显示车辆状态、任务队列、导航请求、路径规划、frontier 前沿点和系统事件。后端通过 WebSocket 持续推送状态快照，所以前端能实时刷新。
```

重点指给老师看：

- 地图覆盖率。
- 行走步数。
- 车辆数量。
- 开放前沿。
- 任务和导航请求。
- 车辆、任务、路径、事件列表。

### 第 2 步：展示运行前配置，约 1 分钟

操作：

1. 地图大小保持 `32 x 24`，分块可以保持默认。
2. 障碍数量建议设置为 `60` 到 `120`。
3. 点击“随机部署”生成障碍。
4. 小车数量建议设置为 `3` 或 `4`。
5. 点击“随机部署车辆”。
6. 任务分配选择 `nearest`。
7. 路径规划选择 `baseline`。

要讲：

```text
运行前可以配置地图大小、障碍、小车数量、任务分配策略和路径规划算法。这里我先用 nearest 策略和 baseline 导航算法作为默认演示，nearest 会给空闲小车分配最近可达的 frontier，baseline 会使用 A* 或时空 A* 进行路径规划。
```

老师可能会问“为什么要先配置再运行”：

```text
因为地图、障碍和车辆部署会影响整个运行状态。系统运行中修改这些配置可能破坏当前任务和路径，所以后端接口在 RUNNING 或 PAUSED 状态下会拒绝修改地图、障碍和车辆数量。
```

### 第 3 步：启动仿真，约 2 分钟

操作：

1. 点击“启动”。
2. 观察车辆开始移动。
3. 指出地图格子从未知变为可通行、已访问或障碍。
4. 指出 frontier、路径、任务、导航请求、路径规划列表开始变化。
5. 等待覆盖率和行走步数增加。

要讲：

```text
启动后，SimulationEngine 每个 tick 会依次调度 Robot、Controller 和 Navigator。Robot 先扫描周围环境并上传地图 patch，同时发现 frontier；Controller 读取黑板中的 OPEN frontier 和空闲车辆，创建任务和导航请求；Navigator 领取请求后规划路径，把路径写回黑板；Robot 再按 pathQueue 执行路径。前端通过 WebSocket 实时接收黑板快照并刷新 3D 地图。
```

重点展示：

- 车辆位置在变化。
- 未知区域逐渐被探索。
- frontier 数量会出现和变化。
- task / navigation request / navigation plan 会产生。
- 系统事件会记录策略切换、运行状态、路径规划等信息。

### 第 4 步：展示暂停、继续、重置，约 1 分钟

操作：

1. 点击“暂停”。
2. 说明车辆停止移动，状态冻结。
3. 再点击“继续”。
4. 点击“重置”，说明状态重新初始化。

要讲：

```text
这些控制按钮对应后端的 `/control/start`、`/control/pause`、`/control/resume`、`/control/reset` 接口。前端只是发送控制命令，真正的状态变化写入 Redis 黑板，再通过快照返回到前端。
```

### 第 5 步：展示算法切换，约 2 分钟

操作：

1. 先点击“停止”或“重置”，不要在运行中切换。
2. 把任务分配策略切换为 `greedy`，点击“应用”。
3. 点击“启动”，观察运行。
4. 再停止，切换为 `low_mdp`，点击“应用”并启动。
5. 如果时间充足，再演示路径规划从 `baseline` 切到 `cbs`。

要讲：

```text
系统把任务分配策略和路径规划算法做成可切换配置。nearest 更重视距离，greedy 更重视 frontier 附近的未知区域收益，low_mdp 则让小车跳过全局任务分配和导航规划，直接基于局部 MDP 做下一步决策。路径规划方面，baseline 适合逐个请求规划，CBS 适合多车同时规划并处理路径冲突。
```

提醒：

- 切换策略前先停止或重置。
- 现场主展示建议用 `nearest`、`greedy`、`low_mdp`、`baseline`、`cbs`。

## 6. 推荐答辩讲解主线

### 6.1 功能主线

```text
功能上，系统支持地图和障碍配置、小车随机部署、任务策略选择、路径算法选择、启动暂停重置、3D 地图实时展示、车辆状态展示、任务队列展示、导航请求和路径计划展示、事件日志展示。
```

### 6.2 架构主线

```text
架构上，系统使用前端、FastAPI、Redis 黑板、Robot、Controller、Navigator 分层设计。前端不直接控制小车，只发送控制命令；Robot、Controller、Navigator 之间也不直接调用，而是通过 Redis 黑板读写共享状态，从而降低耦合。
```

### 6.3 算法主线

```text
算法上，Controller 负责决定哪辆车去哪个 frontier，支持 nearest 和 greedy；Navigator 负责路径规划，支持 A*、时空 A* 和 CBS；Robot 负责执行路径、扫描环境、发现 frontier，在 low_mdp 模式下还可以直接基于局部价值迭代进行自主探索。
```

### 6.4 可视化主线

```text
可视化上，前端使用 WebSocket 接收后端快照，通过 Three.js 渲染 3D 地图、车辆模型、frontier、路径和图层，并同步展示覆盖率、步数、任务、导航请求和事件日志。
```

## 7. 重点亮点

验收时可以突出这些亮点：

1. 黑板架构低耦合：组件只读写 Redis，不直接互相调用。
2. 多车协作探索：多辆车能同时探索未知地图，提高覆盖效率。
3. 可切换算法：任务分配和路径规划都能在前端切换。
4. 实时可视化：WebSocket 快照推送，3D 地图实时刷新。
5. 支持重规划：小车遇到真实障碍时可以上报并触发重新规划。
6. 分布式低耦合：FastAPI、Controller、Navigator、Robot 通过 Redis 黑板通信，可以部署在两台电脑上。
7. 有算法整理和流程图设计稿：不是只做界面，也有明确的软件结构和算法说明。

## 8. 老师可能问的问题和回答

### Q1：为什么要用 Redis 黑板？

```text
Redis 黑板作为共享状态中心，可以让 Robot、Controller、Navigator 不直接互相调用。每个组件只关心自己要读写的数据，比如地图、车辆、frontier、任务、导航请求和路径计划。这样组件之间耦合更低，将来把 Robot 或 Navigator 拆到另一台电脑上也更容易。
```

### Q2：Robot、Controller、Navigator 分别负责什么？

```text
Robot 负责执行和感知，包括扫描地图、上传地图 patch、发现 frontier、执行路径和更新车辆状态。Controller 负责任务分配，决定哪辆空闲车去哪个 frontier。Navigator 负责路径规划，把导航请求转换成可执行路径，并把路径计划写回黑板。
```

### Q3：frontier 是什么？

```text
frontier 是已知可通行区域和未知区域的边界点。小车扫描后会找到这些边界点，Controller 再把 frontier 分配给空闲车辆，这样车辆就会不断向未知区域推进。
```

### Q4：nearest 和 greedy 有什么区别？

```text
nearest 会优先选择真实路径距离最近、可到达的 frontier，稳定且容易解释。greedy 会给 frontier 打分，未知区域收益越高越优先，同时考虑距离惩罚，所以它更偏向探索收益。
```

### Q5：low_mdp 模式有什么特点？

```text
low_mdp 模式下，小车不依赖 Controller 分配目标，也不依赖 Navigator 规划全局路径，而是在局部已知区域内做价值迭代，根据未知格子奖励和多车斥力选择下一步。这是一种更去中心化的局部自主探索策略。
```

### Q6：CBS 是做什么的？

```text
CBS 是 Conflict-Based Search，用于多车联合路径规划。它会先给多辆车各自规划路径，再检测同一时间占用同一格子或对向交换位置的冲突。如果有冲突，就添加约束并重新规划。
```

### Q7：前端怎么做到实时刷新的？

```text
后端提供 `/ws` WebSocket 接口，会不断把黑板快照推送给前端。前端收到快照后合并地图增量 mapDelta，并刷新 3D 地图、车辆、路径、frontier、任务和事件面板。
```

### Q8：这个项目是分布式的吗？

推荐回答要稳一点：

```text
当前验收版本支持两种运行方式：单机演示可以用内嵌 SimulationEngine 跑完整流程；分布式演示可以把前端、FastAPI 和 Redis 放在电脑 A，把 Controller、Navigator 和 Robot fleet worker 放在电脑 B。两种方式使用同一套 Redis 黑板状态模型。
```

### Q9：如果 Redis 挂了会怎么样？

```text
Redis 是黑板状态中心，如果 Redis 不可用，后端健康检查会失败，系统状态无法正常读写。验收时可以通过 `/health/redis` 查看连接状态。实际部署时可以用 Docker restart、持久化卷和运行监控来提高可用性。
```

### Q10：用户分级和回放功能怎么讲？

```text
用户分级和回放功能已经做了用例图设计稿，位于 `图/用户分级用例图设计稿.md`。当前核心验收主线是多小车协作巡检仿真，用户分级和回放可以作为后续软件构造扩展：超级管理员管理账号，运行员配置并启动仿真，分析员查看历史回放和对应数据。
```

1. 浏览器是否打开的是后端地址，不是直接打开本地 HTML 文件。
2. 后端终端是否还在运行。
3. 页面连接状态是否显示 WebSocket 已连接。
4. 刷新浏览器页面。

### 10.5 切换算法失败

常见原因：

- 系统正在 RUNNING。
- 后端接口要求先暂停或停止再切换。

处理：

1. 先停止或重置。
2. 切换到 `nearest`、`greedy` 或 `low_mdp`。
3. 点击应用。
4. 再启动。

## 13. 现场不要踩的坑

1. 不要在老师面前临时安装 Docker 或 Python。
